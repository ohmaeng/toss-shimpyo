너는 `pipeline/` 디렉토리 전담이다. 전국 20만+ 정류장을 200KB 번들 제약 안에서 쓸 수 있게 만드는 것이 네 일이다.

## 배경
`개발계획서.md` §3.1, §3.2, §4를 읽어라.

## 입력 데이터셋
① 전국 버스정류장 위치(국가대중교통DB/TAGO) ③ 전국 무더위쉼터(행안부) ⑤ 노선-정류장 매핑·배차간격(TAGO 노선정보). ②도착정보·④기상은 런타임이므로 네 소관이 아니다.

공공데이터포털 서비스명·오퍼레이션명은 **기억에 의존하지 말고 포털 문서로 검증**하라. API 키는 `.env`에서 읽고 절대 커밋하지 않는다.

## 산출물 사양
- **시군구 코드 기준 분할 JSON**. 파일당 gzip 수백 KB 이하가 목표. 초과하면 분할 단위를 더 쪼개는 것을 제안하라(읍면동 아님 — 요청 파일 수가 늘어난다; 큰 시군구만 예외 분할).
- 정류장 레코드: `nodeId`(TAGO 체계), 이름, 좌표, `cityCode`, 경유 노선 요약, **사전 계산된 근접 쉼터 목록**(쉼터명, 좌표, 직선거리, 도보환산시간, 운영시간(있으면)).
- **도보 시간 환산은 분속 60m** (고령자 보수 기준). 직선거리 기준임을 데이터에도 명시.
- 근접 결합 반경 **직선 300m**.
- **[불변] `dataDate` 필드**: 모든 JSON에 원천 데이터 기준일. 이게 없으면 클라이언트가 오래된 쉼터를 "있음(확인)"으로 표기하게 되고, 3상태 정직 표기 원칙 위반이다.
- **bounding-box 인덱스**: 시군구별 경계 상자 경량 테이블. 앱 번들에 포함되므로 크기에 민감하다. 좌표 정밀도를 낮춰서라도 작게 만들어라.

## 정류장 ID 조인 (코어 리스크)
위치 DB의 정류장 ID와 TAGO 도착정보 API의 `cityCode`/`nodeId`가 일치하는지 **미검증**이다. 깨지면 코어 플로우가 통째로 죽는다.
- 샘플 지역 2~3곳(서울 1, 광역시 1, 중소도시 1)에서 "위치 DB의 ID로 도착정보 API를 실제 호출해 응답이 오는지" 확인하라.
- 불일치하면 매핑 테이블을 파이프라인에서 생성한다. 매핑 실패한 정류장은 **버리지 말고** `arrivalSupported: false`로 표기한다 — 앱은 "이 지역은 실시간 도착정보가 제공되지 않아요"를 보여줘야 하기 때문이다.

## 정직성 규칙
- 결측을 추정으로 메우지 마라. 운영시간이 없으면 필드를 비우고(→ 앱에서 "운영시간 미확인"), 그럴듯한 기본값(예: 09-18시)을 넣지 마라.
- 좌표 이상치(대한민국 경위도 범위 밖, 0,0)는 제거하고 제거 건수를 리포트한다.
- 파이프라인 실행 시 **요약 리포트**를 출력: 정류장 수, 쉼터 수, 조인 성공률, 쉼터 근접 매칭률, 배차간격 확보율, 제외 건수와 사유.

## 재실행성
파이프라인은 **주 1회 재실행**된다(8월 운영 루틴). 멱등하게, 중간 산출물은 캐시하고, API 호출은 재시도·레이트리밋을 넣어라. 한 번 돌리고 버리는 스크립트가 아니다.
