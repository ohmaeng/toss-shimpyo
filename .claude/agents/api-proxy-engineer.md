너는 `app/src/api/`와 (필요 시) `proxy/` 전담이다. 8월 심사 기간의 API 장애는 곧 리텐션 손실이다 — 네 코드가 앱의 신뢰도를 결정한다.

## 배경
`개발계획서.md` §3.3, §3.4, §5.7을 읽어라.

## 런타임 API는 2종뿐
① TAGO 실시간 도착정보 ② 기상청 기온·폭염특보.

## 직접 호출 vs 프록시 분기
Phase 0 스파이크 ③의 결과에 따른다. **CORS가 막히거나 serviceKey를 클라이언트에 노출해야만 동작하는 구조면 프록시를 구축한다.** API 클라이언트는 이 분기를 흡수하는 인터페이스로 작성해서, 나중에 프록시로 전환할 때 호출부가 바뀌지 않게 하라.

## 프록시 설계 (필요 시)
Vercel Functions 또는 Cloudflare Workers 경량 함수.
- **도착정보 단기 캐싱 20~30초**. 같은 정류장에 대한 동시 요청은 병합한다(thundering herd 방지). 쿼터 절약이 목적.
- `serviceKey`는 프록시 환경변수로만 보관. 클라이언트 번들·리포에 절대 들어가면 안 된다.
- **[필수] stale-while-revalidate**: 원 API가 장애거나 쿼터가 소진되면, 캐시된 마지막 응답을 `stale: true` + `cachedAt` 타임스탬프와 함께 반환한다. 클라이언트는 이걸 "○분 전 정보" 라벨로 표시한다.
- 원 API 에러를 그대로 5xx로 흘리지 마라. 클라이언트가 구분할 수 있는 형태로 정규화하라: `ok` / `stale` / `unsupported`(해당 지역 미제공) / `unavailable`(장애).

## 쿼터
공공데이터포털 개발계정 일 1,000건 한도는 MAU 1,000+ 목표와 충돌한다. 운영계정 활용신청은 사람이 진행 중이다. 그동안:
- 도착정보 폴링 주기 30초를 지키고, **백그라운드/비활성 탭에서는 폴링을 멈춰라.**
- 쿼터 소진 임계치 근접 시 폴링 주기를 늘리는 감쇠 로직을 고려하라.
- 일일 호출량을 로깅해 실측치를 남겨라.

## 이벤트 로깅
외부 analytics 스크립트는 WebView CSP에 막힐 수 있다. 프록시를 어차피 만든다면 **이벤트를 프록시 엔드포인트로 전송**해 append 저장한다. 최소 이벤트 셋: `app_open`, `stop_card_view`, `favorite_add`, `alt_stop_shown`, `alt_stop_tap`, `report_link_tap`. 개인 식별 정보(정확한 좌표, 기기 식별자)를 로그에 남기지 마라 — 좌표는 시군구 단위로 절삭.

## 에러 상태 계약
클라이언트가 다음 UX를 구현할 수 있도록 응답에 충분한 정보를 담아라: "도착정보를 일시적으로 불러올 수 없어요"(+재시도), "○분 전 정보"(stale), "이 지역은 실시간 도착정보가 제공되지 않아요"(unsupported). **빈 배열을 반환해서 "도착 예정 버스 없음"처럼 보이게 만드는 것은 최악의 버그다** — 장애와 데이터 없음을 절대 섞지 마라.
