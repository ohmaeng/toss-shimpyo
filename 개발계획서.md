## 1. 제품 개요

### 1.1 한 줄 정의

**"버스가 언제 오는지"에 "기다리는 동안 어디가 시원한지"를 얹은 여름 버스 대기 지도.**

지도에서 내 주변 버스정류장과 무더위쉼터가 함께 보이고, 정류장을 탭하면 **실시간 도착정보 + 도보 O분 거리 시원한 대기 장소 + 폭염 정보**가 한 카드에 뜬다. 국내 주요 지도·버스 앱에 없는 정보 조합(2026.7 기준).

### 1.2 출품 컨텍스트

| 항목 | 내용 |
|---|---|
| 대회 | 토스 앱인토스 7월 바이브코딩 챌린지 (주제: "미니앱과 함께하는 여름") |
| 제출 마감 | **2026-07-29** 첫 앱 번들 제출 (외부 고정. 콘솔 심사 소요일만큼 역산 필요 — §8) |
| 1차 심사 | **8/1~8/26 실사용 지표(AU 등)** — 출시 후 8월 한 달이 심사 기간 |
| 최종 심사 | 완성도 · 테마 적합성 · UX |
| 팀 | 3인 (기획/발표 — 데이터 분석 — 개발), 바이브코딩 전제 |
| 범위 | **전국** (특정 지역 한정 아님) |

핵심 함의: 7/29는 시작점이고 진짜 목표는 **8월에 유저가 매일 다시 여는 것**이다. 실시간 도착정보가 코어인 이유가 이것이다(무더위쉼터 정보만이면 1회성 앱).

### 1.3 [불변 규칙] 3상태 정직 표기 원칙

모든 시설·환경 정보는 다음 3단계로만 표기한다. 데이터로 확인 안 된 것을 단정하지 않는다.

- **있음(확인)** — 데이터로 존재가 확인됨
- **없음(확인)** — 데이터로 부재가 확인됨
- **미확인** — 데이터가 없거나 판단 불가

이 원칙은 UI 문구, 대안 정류장 판정 로직, 엣지케이스 처리 전반에 적용된다. 위반 0건이 성공 기준 중 하나다.

## 2. 플랫폼 & 기술 스택 (확정)

### 2.1 확정 스택

| 레이어 | 선택 | 비고 |
|---|---|---|
| 플랫폼 | **Apps in Toss WebView SDK** | Granite(RN)는 지도 JS SDK 사용 불가로 배제. WebView에서 지도가 전부 실패할 때만 재검토 |
| 스캐폴드 | `npx create-ait-app` | WebView 템플릿 선택 |
| 프론트엔드 | **React + TypeScript + Vite** | |
| 지도 | Kakao Map JS SDK → Leaflet+타일 → 리스트 UI (§2.3 폴백 체인) | Phase 0에서 확정 |
| 데이터 파이프라인 | Python (pandas), 빌드 타임 실행 | 클라이언트 번들과 별개 리포 디렉토리 (`/pipeline`) |
| 정적 데이터 호스팅 | Vercel 또는 Cloudflare Pages 정적 호스팅 (CDN) | 도메인 화이트리스트 통과 확인 필요 |
| 서버리스 프록시 (조건부) | Vercel Functions 또는 Cloudflare Workers | Phase 0에서 필요 여부 판정 (§3.3) |

### 2.2 플랫폼 제약 (앱인토스 요건)

1. **번들 사이즈**: ~200KB 수준 권장. → 전국 정류장 데이터(20만+)는 번들에 절대 포함 불가. 시군구 분할 JSON을 원격 로드 (§3.2). 지도 SDK도 외부 스크립트 로드.
2. **위치**: 앱인토스 브릿지 API `getCurrentLocation` 사용 (브라우저 geolocation API가 아님 — SDK 문서 확인). 권한 거부 플로우 필수 (§5.7).
3. **도메인 화이트리스트/CSP**: 외부 리소스(지도 SDK, 타일서버, CDN JSON, 프록시, API)는 콘솔에 도메인 등록이 필요할 수 있음. **사용하는 모든 외부 도메인 목록을 개발 중 지속 관리**하고 Phase 0에서 등록 절차 확인.
4. **앱 이름 규칙**: 한국어 앱 이름과 코드의 `appName`이 콘솔 등록 정보와 일치해야 함. 앱 이름: **"쉼표 정류장"** (변경 시 콘솔과 동시 변경).
5. **문서**: developers-apps-in-toss.toss.im — WebView 가이드, 브릿지 API 레퍼런스, 배포 절차. 구현 전 최신 문서 확인 필수(변경 잦음).

### 2.3 지도 렌더링 — 폴백 체인 [불변 규칙: 순서 고정]

**알려진 함정 (2026-07-09 리뷰에서 확인)**: 카카오맵 JS SDK가 토스 WebView에서 무한 로딩된 커뮤니티 실사례가 있다(`window.kakao.maps` 미로드). 지도 SDK는 geometry·marker 등 모듈을 CDN에서 **동적 fetch**하므로, 해당 CDN 도메인까지 화이트리스트에 포함되는지가 관건이다.
참고: techchat-apps-in-toss.toss.im/t/js-sdk-webview/3859, /t/app-1/4125

**폴백 체인**: ① Kakao Map JS SDK → ② Leaflet + 타일서버(OSM 또는 VWorld) → ③ 리스트 UI (지도 없음)

**스파이크 통과 기준**: 기본 지도 로드가 아니라 **실기기에서 마커 30개 렌더링 + 팬/줌 + 마커 탭 이벤트까지 성공**해야 통과. 기본 지도만 뜨는 것은 통과가 아님(동적 모듈 로드 함정 때문). 실패 판정 즉시 다음 단계로.

지도 코드는 폴백을 전제로 **어댑터 인터페이스**로 감싼다: `MapAdapter { init, setCenter, addMarkers, onMarkerTap, panTo }` — Kakao/Leaflet/리스트 구현체 교체가 화면 코드에 영향 없도록.

## 3. 아키텍처

### 3.1 전체 데이터 흐름

```
[빌드 타임 — 주 1회 재실행]
공공데이터포털 (정류장 위치 DB, 무더위쉼터, 노선-정류장 매핑·배차간격)
        │
        ▼
Python(pandas) 파이프라인
  - 정제 · 좌표 검증
  - 정류장↔쉼터 근접 결합 (직선 300m, 사전 계산)
  - 시군구 코드 기준 분할 + 데이터 기준일 스탬프
        │
        ▼
시군구별 JSON 파일 ──▶ CDN/정적 호스팅
경량 bounding-box 인덱스 ──▶ 앱 번들에 포함

[런타임]
클라이언트 ──현재 좌표──▶ bounding-box 인덱스 ──▶ 해당 시군구 JSON 1~2개만 fetch
클라이언트 ──정류장 ID──▶ (프록시) ──▶ TAGO 실시간 도착정보 API
클라이언트 ──좌표──▶ (프록시) ──▶ 기상청 기온·폭염특보 API
클라이언트 ──이벤트──▶ (프록시 또는 콘솔 지표) 로깅
```

### 3.2 정적 데이터 전략

- **분할 단위**: 시군구 코드. 파일당 gzip 기준 수백 KB 이하 목표.
- **인덱스**: 시군구별 bounding box 테이블(경량)을 번들에 포함. 현재 좌표 → 시군구 매핑 → 해당 파일만 요청. 경계 근처에서는 인접 시군구까지 최대 2파일.
- **파일 내용(정류장 레코드)**: 정류장 ID(TAGO nodeId 체계), 이름, 좌표, cityCode, 경유 노선 요약, **근접 쉼터 목록(사전 계산: 쉼터명·좌표·직선거리·도보환산시간·운영시간(있으면))**.
- **[리뷰 반영] 데이터 기준일 스탬프**: 모든 JSON에 `dataDate`(원천 데이터 기준일) 필드 포함, 쉼터 카드에 "데이터 기준: YYYY.MM" 표기. 파이프라인은 **주 1회 재실행**(8월 운영 루틴). 스탬프 없이 오래된 쉼터를 "있음(확인)"으로 표기하는 것은 3상태 원칙 위반이다.

### 3.3 런타임 API & 조건부 프록시

런타임 API는 2종뿐이다: ① TAGO 실시간 도착정보 ② 기상청 기온·폭염특보.

- **프록시 필요 판정 (Phase 0)**: TAGO·기상청 API가 WebView에서 CORS로 직접 호출 가능한지 + serviceKey 노출 허용 여부 확인. 사전 조사로는 결론 불가 — 둘 중 하나라도 문제면 프록시 구축.
- **프록시 설계 (필요 시)**: Vercel Functions/CF Workers 경량 함수.
  - 도착정보 **단기 캐싱 20~30초** (같은 정류장 요청 병합 → 쿼터 절약)
  - serviceKey는 프록시 환경변수로만 보관 (클라이언트 노출 금지)
  - **[리뷰 반영] stale-while-revalidate**: 원 API 장애/쿼터 소진 시 캐시된 마지막 응답을 `stale: true` 플래그와 함께 반환 → 클라이언트는 "○분 전 정보" 라벨로 표시
- **쿼터**: 공공데이터포털 개발계정 일 1,000건 한도는 8월 AU 목표와 충돌 → **운영계정 활용신청을 사람이 선제 진행**(승인 지연 있음, §8 체크리스트).

### 3.4 [리뷰 반영] 이벤트 로깅 경로

D7 재방문율·즐겨찾기 등록률은 자체 로깅이 필요할 가능성이 높다(콘솔 기본 지표로 부족할 수 있음). 외부 analytics 스크립트(GA 등)는 WebView CSP에 막힐 수 있으므로:

- **Phase 0 스파이크에 로깅 경로 검증 포함**: 콘솔 기본 지표 범위 확인 + 외부 analytics 로드 가능 여부 확인.
- 외부 스크립트가 막히고 프록시를 어차피 구축한다면, **이벤트를 프록시 엔드포인트로 전송**해 저장(간단한 append 로그면 충분). 프록시가 없는데 외부도 막히면 이 시점에 로깅용 최소 함수 1개를 추가한다.
- 최소 이벤트 셋: `app_open`, `stop_card_view`, `favorite_add`, `alt_stop_shown`, `alt_stop_tap`, `report_link_tap`.

| ④ | 폭염특보·기온 | 기상청 API | 폭염 배너, 카드 기온 | 런타임 |
| ⑤ | 노선-정류장 매핑·배차간격 | TAGO 노선정보 API | 대안 정류장 4중 조건의 조건 1·4 | 빌드 타임 |

- API 키: **TAGO·공공데이터포털·Kakao 모두 발급 완료** (사용자 보유 — 구현 시작 시 `.env`로 전달받을 것, 코드/리포에 커밋 금지).
- 공공데이터포털 서비스명·엔드포인트는 구현 시 포털 문서로 검증할 것 (참고: 도착정보 `ArvlInfoInqireService`, 정류소 `BusSttnInfoInqireService`, 노선 `BusRouteInfoInqireService` 계열 — 정확한 오퍼레이션명은 문서 확인).
- 춘천 전용 데이터(그늘막·벤치·로드뷰 실사)는 **v1 제외**.

### 4.2 정류장 ID 조인 (코어 리스크)

전국 정류장 위치 DB의 정류장 ID와 TAGO 도착정보 API의 `cityCode`/`nodeId` 체계가 일치하는지 **미검증**. 조인이 깨지면 코어 플로우가 통째로 깨진다. → Phase 0에서 **샘플 지역 2~3곳**(예: 서울 1, 광역시 1, 중소도시 1)으로 "위치 DB에서 뽑은 정류장 ID로 도착정보 API 호출이 실제 응답을 주는지" 검증. 불일치 시 매핑 테이블을 파이프라인에서 생성.

## 5. 기능 명세

### 5.1 화면 구조 (시민용 단일 화면군 — 행정용 대시보드 없음)

```
[메인: 지도 화면]
  ├─ 상단: 폭염특보 배너 (특보 발효 시에만)
  ├─ 지도: 내 주변 정류장 마커 + 무더위쉼터 마커 (구분된 아이콘)
  ├─ 하단 시트: 정류장 카드 (마커 탭 시)
  └─ 검색 바 / 즐겨찾기 탭 진입점

[정류장 카드 (바텀시트)]
  ├─ 정류장명 + 즐겨찾기 토글
  ├─ 실시간 도착정보 (노선별 O분 후, 30초 주기 갱신)
  ├─ 여름 정보: 가까운 쉼터 (이름, 도보 O분, 운영시간, 데이터 기준일) + 현재 기온
  ├─ 대안 정류장 제안 (조건 충족 시에만, §5.3)
  └─ 제보 버튼 (외부 폼 링크)

[즐겨찾기 화면]  정류장 리스트 → 탭 시 카드
[검색 폴백 화면]  위치 권한 거부 시 정류장명·지역 검색
```

### 5.2 코어 플로우 (M1 = 리스트 UI로도 성립해야 함)

위치 → 내 주변 정류장 → 정류장 선택 → 카드(도착정보 + 쉼터 + 기온). **크래시 없이 3초 내 로드**가 성공 기준. 지도 폴백 ③(리스트 UI)이 발동해도 이 플로우는 동일하게 성립하도록, 카드/데이터 로직을 지도와 분리해 구현한다.

### 5.3 대안 정류장 제안 — 4중 조건 [불변 규칙]

다음 **4개 조건을 모두 만족할 때만** "인근의 더 시원한 정류장"을 제안한다. 하나라도 미충족·미확인이면 제안하지 않는다(오제안이 신뢰를 깨는 것이 무제안보다 나쁘다).

1. **동일 노선**이 정차하는 인접 정류장일 것
2. **직선거리 300m 이내** — 도보 시간은 분속 60m(고령자 보수 기준)로 환산, UI에 "직선거리 기준" 병기
3. **시원함 우위가 데이터로 확인**될 것 — v1 판단 근거는 **무더위쉼터 도보시간 단축 하나로 한정**. 그늘·의자 등 시설 데이터는 v1에 없으므로 판정에 사용하지 않음. 미확인 항목은 판정에서 배제.
4. **도보 예상시간 ≤ 해당 노선 배차간격의 1/2** — 배차간격을 확인할 수 없는 노선은 제안 비활성화

일정 슬립 시 이 기능이 1순위 컷 대상이다 (§7.3).

### 5.4 즐겨찾기

로컬 저장(localStorage 또는 앱인토스 스토리지 브릿지 — Phase 0에서 확인). 즐겨찾기 정류장은 앱 진입 시 바로 도착정보 갱신. 등록률 20%+가 지표 목표이므로 카드에서 원탭 등록.

### 5.5 제보 (v1: 외부 폼)

백엔드 없이 외부 폼(구글 폼 등) 링크. 정류장 카드에서 탭 시 정류장 ID를 쿼리 파라미터로 넘겨 폼에 프리필. 제보 검수·반영 절차는 v2(8월)에서 설계.

### 5.6 폭염특보 배너

기상청 특보 API 기준, 현재 위치 관할 구역에 폭염특보 발효 중일 때만 상단 배너 표시. 미발효 시 미표시(항상 떠 있으면 무시된다). 특보 API 실패 시 배너 자체를 숨김(오탐 방지).

### 5.7 엣지케이스 & 에러 상태 UX (전체 목록)

| 상황 | 처리 |
|---|---|
| 위치 권한 거부/실패 | 정류장명·지역 검색 입력으로 폴백. 권한 재요청 안내는 1회만 |
| 반경 300m 내 쉼터 없음 | "반경 300m 내 쉼터 데이터 없음" 정직 표기 (3상태 원칙) |
| 쉼터 운영시간 외 | 운영시간 표시 + 흐림 처리 + "운영시간 아님" 라벨 |
| 쉼터 운영시간 필드 없음 | "운영시간 미확인" 라벨 (흐림 처리 없음 — 3상태 원칙) |
| TAGO 미제공 지역/노선 | "이 지역은 실시간 도착정보가 제공되지 않아요" + 경유 노선 정보만 표시 |
| **[리뷰 반영] API 장애·쿼터 소진·프록시 다운** | 카드에 "도착정보를 일시적으로 불러올 수 없어요" + 재시도 버튼. 프록시에 stale 캐시가 있으면 "○분 전 정보" 라벨로 대체 표시. **8월 심사 기간의 장애가 곧 리텐션 손실이므로 이 상태의 UX 완성도는 필수 범위** |
| 시군구 JSON 로드 실패 | 재시도 1회 → 실패 시 "데이터를 불러올 수 없어요" + 재시도 버튼 (빈 지도를 정상인 척 보여주지 않음) |
| 오프라인 | 전역 오프라인 배너 + 마지막 캐시 데이터에 "오프라인 — ○분 전 정보" 라벨 |

## 6. 디자인 가이드

### 6.1 방향

- **토스 안의 앱**이다: 토스 UI 톤(넉넉한 여백, 카드·바텀시트 중심, 큰 타이포, 절제된 색)과 이질감 없게. 과한 장식·그라데이션 배제.
- **첫 화면에서 "여름 앱"임이 즉시 인지**되어야 한다(심사 기준). 수단: 폭염특보 배너, 쉼터 마커의 시원한 색, 카드의 기온 표시. 배경 전체를 여름 테마로 칠하는 방식은 지양.
- **정보 위계**: 카드에서 1순위 도착정보(가장 큰 타이포), 2순위 쉼터 도보시간, 3순위 기온·기타. 유저의 원 질문이 "버스 언제 와"이기 때문.

### 6.2 컬러 (제안 — 구현 시 미세 조정 가능)

| 경고 (폭염) | 오렌지~레드 (예: `#F04452`) | 폭염 배너·특보 라벨 전용. 남용 금지 |

## 7. 개발 단계 (Phase 0 → M4)

### 7.1 Phase 0 — 플랫폼 검증 스파이크 (최우선)

`npx create-ait-app` 스캐폴드 후, **샌드박스가 아닌 실기기에서** 다음 5종을 검증한다. 이 결과가 이후 모든 구조를 확정한다.

| # | 스파이크 | 통과 기준 | 실패 시 |
|---|---|---|---|
| ① | 지도 렌더 (Kakao JS SDK) | 마커 30개 + 팬/줌 + 탭 이벤트 (§2.3) | Leaflet+타일 → 리스트 UI 순 폴백 |
| ② | `getCurrentLocation` 권한 플로우 | 허용/거부 양쪽 동작 확인 | 검색 폴백 UX 비중 상향 |
| ③ | TAGO 직접 호출 | CORS 통과 + 키 노출 정책 확인 | 프록시 구축 확정 (§3.3) |
| ④ | 원격 정적 JSON 로드 | CDN 도메인 화이트리스트 통과 | 호스팅 채널 변경 또는 등록 절차 진행 |
| ⑤ | 정류장 ID ↔ TAGO nodeId 조인 | 샘플 지역 2~3곳에서 도착정보 응답 수신 | 매핑 테이블 파이프라인 추가 |
| ⑥ | 이벤트 로깅 경로 [리뷰 반영] | 콘솔 지표 범위 확인 + 외부/프록시 로깅 가능성 확인 | 프록시 로깅 엔드포인트 추가 |

Phase 0 종료 시 산출물: **확정 아키텍처 메모** (지도 단계, 프록시 유무, 로깅 경로, 화이트리스트 등록 목록).

### 7.2 마일스톤 (순서만 고정 — 날짜는 팀이 정한다)

| 마일스톤 | 범위 | 완료 판정 |
|---|---|---|
| **M1** (=리스트 앱으로도 성립) | 데이터 파이프라인 + 코어 플로우(위치→정류장→카드) + 엣지케이스·에러 상태 UX + 로깅 확정. 프록시 구축(캐싱·stale 포함) | **지도 없이도 코어 플로우가 성립한다** |
| **M2** (=B 완성) | 지도 UX, 대안 정류장 4중 조건, 즐겨찾기, 제보 링크, 폭염특보 배너 | 지도 폴백 어느 단계에서도 앱이 온전하다 |
| **M3** (제출) | 콘솔 제출, 심사 리젝 대응 | 규정 검사 P0 0건 |
| **M4** (운영) | 지표 모니터링, 주기적 개선 배포(파이프라인 재실행 포함), 리텐션 기능 | — |

M1은 M2 없이도 제출 가능한 앱이다. **완벽한 M2보다 제출 가능한 M1이 항상 우선한다.**

### 7.3 컷 규칙 [불변 규칙]

**대안 정류장 4중 조건 + 데이터셋 ⑤(노선 매핑·배차간격)** 가 1순위 컷 대상이다.

지도가 컷 대상이 아니라 대안 제안이 컷 대상인 이유: 지도는 첫인상(테마 심사), 도착정보+쉼터 카드는 재방문(AU 심사)에 직결되지만 대안 제안은 부가 기능이기 때문.

**컷 발동 조건 (둘 중 하나라도 참이면 재논의 없이 M4로 이월):**

1. **배차간격 확보율이 20% 미만** — `python pipeline/build_data.py --probe --city-codes ...` 로 측정한다. 이 경우 4중 조건 #4가 대부분 탈락하므로, 기능이 화면에 뜨지 않는다. 코드가 있어도 없는 것과 같다.
2. M2 잔여 작업이 남은 가용 기간의 절반을 넘는다.

조건 1은 **M1 착수 전에 측정할 수 있다.** 감이 아니라 숫자로 결정하라 — 일정에 쫓겨서가 아니라 데이터가 없어서 자르는 것이다.

### 7.4 리포 구조 (제안)

│       ├── components/ # §6.3 컴포넌트

└── docs/         # 이 문서, 확정 아키텍처 메모, 화이트리스트 목록

## 11. Opus 작업 지침

1. **시작 순서**: 이 문서 정독 → 앱인토스 최신 문서 확인 → `npx create-ait-app` 스캐폴드 → Phase 0 스파이크 6종 (§7.1) → 확정 아키텍처 메모 작성 → M1 착수.
2. **자율 결정 범위**: 상태 관리 라이브러리, 스타일링 방식, 폴더 세부 구조, 컴포넌트 세부 설계, 파이프라인 세부 구현. 단 **[불변 규칙]** 표시 항목(3상태 원칙, 폴백 체인 순서, 4중 조건, 컷 규칙)은 변경 불가.
3. **막히면**: 스파이크 실패는 실패가 아니라 판정이다 — 즉시 폴백 단계로 넘어가고 사람에게 판정 결과만 보고. 사람 확인이 필요한 항목(§8)이 블로커면 그 항목을 명시하고 다른 작업을 병행.
4. **비밀 관리**: API 키는 `.env` + `.gitignore`. serviceKey는 프록시 환경변수에만 존재한다.
