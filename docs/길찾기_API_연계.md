# 쉼표 정류장 — 길찾기 API 연계 결정

> 팀 결론: 대중교통 경로탐색 알고리즘을 새로 개발하지 않는다. 외부 대중교통 길찾기 API가 경로를 계산하고, 쉼표 정류장은 춘천시 정류장 ID·TAGO 실시간 도착정보·대기시설 정보·불편 신고를 결합한다.

## 왜 직접 개발하지 않는가

자체 경로 엔진은 정류장 그래프, 노선 진행방향, 환승, 시간표, 도보 연결, 운행 변경을 계속 최신화해야 한다. 이것은 해커톤의 핵심 차별성이 아니며 기존 전문 API보다 정확하기 어렵다.

우리 개발 범위는 다음과 같다.

| 영역 | 외부 API | 쉼표 정류장 |
|---|---|---|
| 장소검색 | 목적지명→좌표 | 음성·문자 입력과 인식 결과 확인 |
| 대중교통 경로 | 출발·도착 좌표→버스·환승·승차 정류장 | 결과를 고령자용 화면으로 변환 |
| 노선·도착 | TAGO 노선·정류장·도착예정 | 실시간/대체정보·생성시각 표시 |
| 정류장 연결 | ODsay·TAGO 원본 ID | 내부 관리번호·4자리 번호와 검증 매핑 |
| 대기환경 | 춘천시 공식자료 | 승차 지점별 시설상태·출처·확인일 결합 |
| 행정 환류 | 해당 없음 | 음성 불편 구조화·자료 대조·현장확인·시민 갱신 |

> **주석 — “API 정보를 우리 판에 붙이면 되는가?”**  
> 기본 방향은 맞다. 다만 외부 정류장 ID와 춘천시 관리번호·4자리 번호·TAGO nodeId를 잘못 연결하면 맞은편 정류장의 시설과 도착정보가 붙는다. 핵심 개발은 경로 계산이 아니라 정류장 정합성 어댑터, 고령자용 표현, 시설정보 결합, 실패 폴백이다.

## 권장 조합

| 역할 | 1차 선택 | 판단 |
|---|---|---|
| 대중교통 전체 경로 | ODsay 대중교통 길찾기 | 전국 대중교통 길찾기와 Web/Mobile 지원. Basic은 1일 1,000회·6개월 무료이며 운영 확대 시 협의 필요 |
| 노선·경유 정류장 | TAGO 버스노선정보 | 노선·경유 정류장을 제공하지만 임의 출발지·목적지 전체 경로 엔진은 아님 |
| 실시간 도착 | TAGO 버스도착정보 | 정류소ID별 도착예정시간 제공. 내부 ID 연결이 선행되어야 함 |
| 목적지 검색·지도 | 카카오 Local·지도 SDK 등 | 목적지 좌표·화면 지도에 사용. 카카오모빌리티 공개 길찾기를 대중교통 경로 API로 가정하지 않음 |
| 시설정보 | 춘천시 위치·노선·BIT·시설자료 | 승차 지점에 결합하고 출처·기준일을 분리 표시 |

## 실제 처리 순서

```text
공통 QR
→ 목적지 음성 입력·확인
→ 장소검색으로 목적지 좌표 확정
→ ODsay 대중교통 경로 조회
→ 첫 버스 구간의 승차 정류장·노선·방면 추출
→ ODsay 정류장 ID ↔ TAGO nodeId ↔ 내부 관리번호 검증 연결
→ TAGO 도착정보와 춘천시 시설정보 병렬 조회
→ 도보시간 안에 탈 수 있는 후보만 남김
→ 목적지 도착예상시간 순으로 정렬
→ 한 화면에 버스·총 소요시간·도착예정·도보·목적지 도착예상·대기환경 표시
→ 결과 화면에서 불편 신고
```

## 결과 화면 계산·표시 계약

| 항목 | 기준 |
|---|---|
| 총 소요시간 | 승차 정류장까지 도보 + 대기 + 버스 탑승 + 하차 후 도보 |
| 추천 순서 | 가장 먼저 오는 버스가 아니라 목적지 도착예상시간이 빠른 순서 |
| 탑승 가능성 | `버스 도착예정 ≥ 승차 정류장까지 도보시간 + 이동 여유시간`인 후보만 표시 |
| 대표 카드 | 버스번호, 총 소요시간, 버스 도착예정, 타는 곳까지 도보시간, 목적지 도착예상 |
| 보조 정보 | 탑승 정류장명·4자리 번호·방면, 버스 탑승시간, 하차 후 도보, 시설상태 |
| 화면 수 | 결과 한 페이지. 별도 경로 상세 화면은 필수로 만들지 않음 |

이동 여유시간은 임의의 고정 사실로 표시하지 않고 사용자 테스트로 기본값을 정한다. 실시간 도착정보와 경로 예상시간은 각각 출처와 생성시각을 구분하고, 값이 없는 항목을 추정해서 채우지 않는다.

> **주석 — 외부 API를 쓰면 개발이 끝나는 것은 아니다.**  
> API 선정 뒤 첫 검증은 UI가 아니라 춘천 대표 경로 10건의 승차 정류장 ID·좌표·방면을 실제 지도와 대조하는 것이다. 명칭·좌표만 비슷한 정류장은 자동 확정하지 않는다.

## 장애·불일치 해결방안

| 문제 | 처리 |
|---|---|
| ODsay 경로 실패 | 타임아웃 후 오류 안내. 본선 대표 경로는 출처·생성시각이 표시된 캐시로 폴백 |
| TAGO 도착 실패 | 실시간 배지 제거. 근거가 있는 배차 참고만 표시하거나 미제공 처리 |
| 시설자료 없음 | `미확인`으로 표시하고 없음으로 추정하지 않음 |
| 정류장 ID 불일치 | 시설·도착 결합을 중단하고 검수 후보로 저장 |
| 맞은편 혼동 | 노선 진행방향·정차순서·방면을 함께 검증 |
| API 호출량 초과 | 경로 응답 단기 캐시, 동일 요청 중복 제거, 운영 전 요금·협약 검토 |
| 외부 API 종속 | API별 어댑터 인터페이스로 분리해 교체 가능하게 구현 |

## 본선 구현 우선순위

1. ODsay가 춘천 대표 경로를 실제로 반환하는지 10건 검증
2. ODsay 정류장과 TAGO·내부 관리번호 연결 어댑터 구현
3. 목적지 음성→경로→도착→시설→불편 신고 한 사이클 완성
4. 경로·도착·시설 API 각각의 실패 폴백 검증
5. 복잡한 자체 환승 최적화와 자체 경로 엔진은 제외

## 팀 발표 문장

> 쉼표 정류장은 기존 대중교통 API를 다시 만드는 서비스가 아니다. 검증된 경로 결과에 춘천의 승차 지점별 대기환경을 결합하고, 기다리는 시민의 한마디를 AI가 행정의 검증 가능한 과업으로 바꿔 다시 시민 정보로 환류한다.
