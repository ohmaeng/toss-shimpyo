## 1. 완료 기준 (본선 데모 시나리오)

1. (B2C) 정류장 QR 스캔 → 가입 없이 즐겨찾기 등록 + 확인 문구 *(사전 1회 접속된 기기)*
2. (B2C) 목적지 선택 → "앉아서 기다리는 길": 최단/시설우선 토글, 이유 문장(점수 숫자 없음), 미확인 명시
3. (B2C) 최초 접속 후 오프라인 전환 → 전 흐름 폴백 동작
4. (B2G) **1단계: 조사 우선순위** — 수요 실측 × 정보 미확인 기반, "조사할 곳" 목록·CSV
5. (B2G) **로드뷰 조사 결과 반영 라이브 시연** — 조사 CSV 투입 → 파이프라인 재실행 → 미확인 감소 + 2단계 활성화되는 장면
6. (B2G) **2단계: 시설별 설치 검토 우선순위** — 조사에서 '없음' 확인된 정류장만, 벤치/그늘막/조명 별도 표 + 시설별 예산
7. (B2G) 근거 카드(산식·구성요소 실측·출처·기준일) → CSV(재검산 가능)·A4
8. 프리셋 3종 전환 + 정책 시나리오 비교 표("프리셋을 바꿔도 Top10이 얼마나 유지되는가" — "민감도 분석" 지칭 금지)

## 2. 규칙 (2026-07-16 개정 + v3.1 강화)

- 합성 지수는 5조건 하에서만: ①산식 공개 ②구성요소 실측값 병기 ③출처·기준일 배지 ④B2G만 숫자, B2C는 정렬+문장 ⑤**변수 수준 분리 — `확인된 없음(confirmedDeficit)`과 `미확인(unknownRate)`은 절대 한 변수로 합치지 않는다.** 미확인은 조사 트랙에만, 확인된 없음은 설치 트랙에만 쓴다.
- 기존 정직성 규칙 전부 유지(3상태, "현장 확인" 금지, "양방향 합산" 병기, 거짓 "실시간" 금지, 노선 단위 수요 주장 금지).
- 표현 추가: "그늘 4곳"류 단독 사용 금지(→ "확인 4곳, 1,886곳 미확인"). "수혜자/수혜 건수" 사용 금지. 오프라인 주장엔 "최초 1회 접속 후" 조건 병기.

## 3. B2G — 2단계 정책 흐름 (핵심 재설계)

```
[현재 데이터]                [조사]                    [조사 후 데이터]
수요 실측 + 시설 미확인  →  1단계: 조사 우선순위   →  로드뷰/현장 조사(AI 보조)
                             "어디부터 확인할까"        yes/no 확정
                                                        ↓
                             2단계: 설치 검토 우선순위 ← '없음' 확인된 정류장만
                             "벤치는 어디, 그늘막은 어디"  시설별 별도 순위
                                                        ↓
                             시설별 예산 시뮬레이터 → 설치 → 데이터 갱신(선순환)
```

이 구조의 가치: **"미확인은 미확인으로"가 문구가 아니라 업무 흐름 그 자체**가 된다. 현재 no=0건이라는 데이터 한계를 숨기지 않고 "그래서 조사가 1단계"라고 정면으로 설계에 반영.

### 3.1 1단계 — 조사 우선순위 (지금 데이터만으로 성립)

```
surveyPriority = ( w_d·D + w_u·UNK [+ w_p·P] ) / Σ(활성 w)
  D   = 한낮(프리셋별 시간창) 승차 분위수 [0,1]     ← 실측
  UNK = 미확인 시설 수 / 4 [0,1]                   ← "정보 부족도" (결핍 주장 아님)
  P   = 생활지원시설 인접도 [0,1] — **기본 비활성.** 고령자 이동지원 프리셋에서만 활성
        (조사 효율의 핵심은 수요+미확인도. POI는 선택적 정책 가치판단 — Codex 재판정)
```

- 의미: "수요가 많은데 정보가 부족한 곳부터 조사한다" — 미확인을 미확인으로만 사용, 완전히 방어 가능.
- **표현: "점수"가 아니라 "조사 검토 순서".** CSV에는 각 원값과 선정 사유(어느 항이 순위를 끌었는지) 병기.
- **수요 미확인 정류장(223곳 포함)은 이 순위표에 섞지 않는다.** 별도 그룹 "수요·시설 모두 미확인 — 조사 후보(순위 없음)"로 목록 하단 표기.
- 노선 수는 산식에서 제외(상관분석 결과 유의미하면 재도입 검토), 표에는 원값 표시.

### 3.2 2단계 — 시설별 설치 검토 우선순위 (조사 데이터가 활성화)

```
installPriority(시설f) : 대상 = f가 '없음(no, 로드뷰/현장 확인)'인 정류장만
  순위 키 = w_d·D + w_p·P   (시설별 별도 표: 벤치 / 그늘막 / 조명)
```

- 기설치(yes) 정류장은 해당 시설 트랙에서 제외. 미확인은 **대상 아님**(1단계로).
- 현재 no=0 → 본선 전 로드뷰 조사(§6)가 이 단계를 채운다. 데모 시나리오 5가 전/후 변화를 라이브로 보여줌.
- **명칭은 끝까지 "설치 검토 우선순위"("설치 우선순위" 금지).** 모든 행·CSV에 상태 라벨 고정: **"데이터상 설치 검토 후보 · 현장 설치 적합성 미검토"** — 보도폭·도로점용·사유지·지하매설물은 데이터가 없으므로 아는 척하지 않고, 최종 판단은 행정의 현장 검토임을 명시.

### 3.3 가중치 — 프리셋 우선, 슬라이더 보조

- 프리셋 3종이 기본 UI: **폭염 대응형**(D 한낮 강화, P 비활성), **고령자 이동지원형**(P 활성), **이용량 중심형**(D 단독). 프리셋별 가중치 값과 설정 근거 한 줄을 화면에 명시.
- 슬라이더는 "직접 조정" 패널로 보조 제공(행정 담당자의 정책 판단 도구).
- **프리셋별 순위 안정성 표**: 프리셋 3종에서 Top10 진입 빈도(3/3, 2/3…) + 평균·최저·최고 순위. **발표에서 "민감도 분석"이라 부르지 않는다**(세 프리셋은 팀이 정한 세 점일 뿐) — 정확한 명칭은 **"정책 시나리오 비교"**. 시간 여유 시 무작위 가중치 조합 N개로 진짜 민감도(순위 변동 폭) 확장 가능(선택).

### 3.4 예산 시뮬레이터 — 시설별

- 트랙별(벤치/그늘막/조명) 단가·기준·출처를 화면에 명시(단가는 "가정, 편집 가능" 라벨. 조달 단가 확보 시 교체).
- 결과 표기: "대상 정류장 N곳 / **표본기간(2025.6, 4일) 한낮 승차 합계 M건** (양방향 합산, 수요 미확인 K곳 제외)". "수혜"·"일평균" 표현 금지(공식 환산 근거 확보 전).

## 4. B2C — "앉아서 기다리는 길"

- 정렬 코어 `comfortScore(stop)` = `(seatYes + shadeYes + night?lightYes) / 활성가중치합` — **괄호 명시: 분자 전체를 활성 가중치 합으로 나눈다.** yes만 가점 — **"확인된 시설 존재 우선"이며 미확인을 시설 없음으로 단정하지 않는다**("확실성 선호" 표현 금지 — no도 확실하지만 선호되지 않으므로 부정확). B2G 산식 사용 금지(의존성 테스트). 화면·발표에서 "편안함 점수" 설명 금지.
- **사용자 선택 토글**: "가까운 순" / "시설 확인된 곳 우선"(기본). 부제 문구: "확인된 시설이 있는 길을 우선 보여드려요".
- **이유 문장은 시설별로 구체화** — 명칭("앉아서 기다리는 길")과 결과가 어긋나지 않게:
  - 의자 확인: "앉아서 기다릴 수 있어요 (벤치대장 기준)"
  - 그늘만 확인: "그늘이 확인된 정류장이에요"
  - 조명만 확인(야간): "조명이 확인된 정류장이에요"
  - 시설 미확인: "시설 정보가 아직 확인되지 않았어요"
  의자 미확인 카드에 "앉아서" 문구 사용 금지(테스트 강제). 점수 숫자 비표시(테스트 강제).
- **실효 커버리지 측정(본선 전 필수)**: 전 정류장 기준 (a)시설우선 정렬이 최단 정렬과 다른 결과를 내는 비율 (b)확인 시설 후보가 존재하는 비율 (c)추가 도보시간 분포 — 결과를 근거 카드/발표에 공개. 낮으면 낮다고 말하고 "조사가 커버리지를 늘린다" 서사로 연결.
- 정류장 QR: v3 그대로(카드 버튼·포스터 삽입·등록 확인 문구·배포 URL 선고정).

6. 표본 4일의 요일 구성·특이사항 기록
7. 데이터 기준일 불일치표(수요 2025.6 vs 노선 2026.3 vs 시설 2026.6)

**사용자 검증(본선 전, 사람):** 60대+ 3~5명 — QR 등록 성공률, 목적지 선택 완료율/소요시간, 오조작 횟수, "앉아서 기다리는 길" 의미 이해율, 글자·버튼 만족. 결과와 개선 반영 내역을 발표 1장으로.

**AI 판독 효율 측정:** 20~30곳에서 사람 단독 vs AI 보조 처리시간, 시설별 정밀도/재현율, 판별불가율, 사람 수정률. **측정 전 발표 문구는 "AI 보조로 판독 초안·문서화를 표준화" 까지만.**

**오프라인 E2E 매트릭스:** {최초 접속/재방문} × {브라우저/설치 PWA} × {완전 오프라인/느린 네트워크} × {새로고침/직접 URL/QR 진입}. "QR 최초 진입 + 오프라인 = 실패"를 알고 있는 한계로 문서화, 발표 데모는 사전 접속 기기 사용.

## 6. AI 로드뷰 판독 (본선 전 실행)

사람 캡처 → AI 초안 → 사람 확정 → `roadview_survey_template.csv` → 파이프라인. 대상 목록 = §3.1 조사 검토 순서 상위 150. 효율 측정(§5) 포함. **이 조사가 2단계를 활성화**하는 것이 본선 데모의 척추. 로드뷰 캡처 발표 사용 시 출처(카카오맵·촬영연월) 표기.

**판정 기준 (로드뷰 1장으로 '없음' 확정의 한계 — Codex 재판정 5):**
- **yes**: 영상에서 시설이 명확히 확인될 때만
- **no**: 정류장 전체 조사 범위가 충분히 보이고 시설 부재를 확인할 수 있을 때만 (촬영 방향 밖·가림·촬영일 이후 설치 가능성을 검수자가 판단)
- **unknown**: 시야·촬영일·방향·화질 때문에 부재를 확정할 수 없으면 유지
- **조명은 보수적으로 unknown 기본**(주간 로드뷰로 야간 조명 판별 곤란) — AI가 no로 판정해도 변환기가 "미확인"으로 강제, 사람이 명시적으로만 변경 가능
- 촬영연월을 capturedAt에 기록(오래된 로드뷰의 시차 한계 공개)

**데모 서사 (연출 정확성):** 미리 조사한 CSV 투입을 "현장조사 라이브"처럼 보이게 하지 않는다. 정확한 문장: *"본선 전에 사람이 검수한 로드뷰 조사 결과를 행정 조사 입력 형식으로 투입하면, 설치 검토 목록이 자동 생성됩니다."* 전후 수치 4종 표시: 투입 전(no=0·후보 0곳) / 조사 확정본(N곳: yes A·no B·unknown C) / 투입 후(시설별 후보 수) / 제외된 unknown 수.

## 7. P2 실험 격리 — 익명 카운터·AI 브리핑

- 본선 필수 기능 아님. 착수 조건: P0·P1 전부 완료 + 팀 합의.
- 카운터 도입 시 정직 표기: "클라이언트는 식별자를 보내지 않으며, **호스팅 인프라의 접속 로그에 IP가 남을 수 있어** 로그 보존을 최소화/비활성화한다" — "IP 미저장"을 서버까지 보장하는 것처럼 말하지 않는다. 옵트인 UI·보존기간·중복이벤트 정책 정의 전에는 배포 금지.
- AI 브리핑은 사전 생성 정적 JSON만(클라이언트 LLM 키 금지). 발표에서는 로드맵("이렇게 확장 가능")으로 언급.

## 8. 접근성 (정정)

- **둘 다**: 타이핑 없이 탭/스캔만으로 핵심 흐름 완결(고령자) **그리고** 키보드만으로도 전 핵심 흐름 조작 가능(WCAG 2.1.1). 포커스 순서·표시, 스크린리더 레이블, 색상 외 상태 표현(아이콘+텍스트), 지도 없이 목록으로 동일 정보 접근, 명암비 실측, `prefers-reduced-motion`.

## 9. 보안·배포 (v3 유지)

공개 배포 무키 빌드 / 배포 URL 선고정 후 QR 포스터 / stops.json 계약 불변 / 클라이언트 번들에 LLM·과금 키 금지.

## 9-1. 실행 중단 조건 (Codex 최종 승인 조건 — 계획서 🛑 절과 동일)

① unknown이 설치 검토 후보에 진입 ② 미검수 AI 초안이 no 생성 ③ 데이터 검증·사용자 테스트 완료 전 P2 착수 — 셋 중 하나라도 발생하면 즉시 중단·보고. 기능 추가보다 **전후 수치 라이브 데모 + 검증 리포트 + 사용자 테스트의 안정적 완성**이 우선.
