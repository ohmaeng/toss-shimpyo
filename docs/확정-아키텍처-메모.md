# 확정 아키텍처 메모 (Phase 0 결과)

작성: 2026-07-10 · 근거: `@apps-in-toss/web-framework@2.10.5` 패키지의 타입 정의 직접 검증 + 공식 문서
검증 방법: npm 설치 후 `.d.ts` 원문 확인. **문서보다 패키지가 정확하다** — 아래 항목은 문서가 아니라 실제 타입에서 읽었다.

## 1. 계획서에서 틀렸던 것

| 계획서 기술 | 실제 | 영향 |
|---|---|---|
| `npx create-ait-app` | **`npx ait init`** (`create-ait-app`은 v0.1.3의 다른 패키지) | 스캐폴드 명령 수정 |
| 설정 파일 미상 | **`granite.config.ts`** + `defineConfig` from `@apps-in-toss/web-framework/config` | — |
| 자체 이벤트 로깅 경로 불명, 외부 analytics가 CSP에 막힐 위험 | **SDK에 `Analytics` 내장** (`screen`/`impression`/`click`) | 외부 analytics 불필요. 스파이크 ⑥ 해소 |

## 2. 확정된 브릿지 API (실제 시그니처)

```ts
import {
  getCurrentLocation, Accuracy,   // getCurrentLocation.getPermission() / .openPermissionDialog()
  getNetworkStatus,               // 'OFFLINE' | 'WIFI' | '2G' | ... → 오프라인 배너
  openURL,                        // 제보 폼 링크
  Storage,                        // getItem/setItem/removeItem → 즐겨찾기 영속화
  Analytics,                      // screen/impression/click
  requestNotificationAgreement,   // 푸시 동의 (계획서 리스크 7 → 지원됨)
} from '@apps-in-toss/web-framework';
```

- `getCurrentLocation({ accuracy: Accuracy.Balanced })` → `Promise<{ coords: { latitude, longitude, ... }, timestamp }>`
- **권한 3상태**: `'notDetermined' | 'denied' | 'allowed'` — `getPermission()`으로 조회, `openPermissionDialog()`로 재요청. 계획서의 "재요청 안내 1회만"을 정확히 구현 가능.
- `granite.config.ts`의 `permissions`에 **`'geolocation'`** 을 선언해야 한다 (`PermissionName = 'clipboard' | 'contacts' | 'photos' | 'geolocation' | 'camera' | 'microphone'`).

**스파이크 ② 해소**: 위치 권한 플로우는 SDK가 3상태를 그대로 노출한다.
**스파이크 ⑥ 해소**: `Analytics` 내장 → 프록시 로깅 엔드포인트는 **선택**이 됨. (다만 D7 코호트 분석을 위해 프록시 로깅도 병행 구현했다 — 콘솔 지표의 범위가 아직 미확인이기 때문.)

## 3. 프록시: **필수로 확정** (스파이크 ③)

계획서는 "CORS가 막히거나 키 노출이 문제면 프록시"라는 조건부였다. **조건부가 아니다 — 프록시는 필수다.**

이유: 공공데이터포털 API는 `serviceKey`를 쿼리스트링으로 요구한다. 클라이언트에서 직접 호출하면 **번들과 네트워크 탭에 키가 그대로 노출**된다. CORS 허용 여부와 무관하게 키 노출만으로 프록시가 강제된다. CORS를 실기기에서 테스트할 필요조차 없다.

→ `proxy/` 구현. Vercel Functions. 도착정보 25초 캐싱 + stale-while-revalidate + 동시요청 병합.

## 4. 지도: 폴백 체인 유지, 어댑터로 격리

Kakao SDK를 실기기에서 검증하는 것은 **사람의 실기기 테스트가 필요한 유일한 항목**이다(§8). 코드는 3단계 폴백을 모두 구현했고, 어댑터 뒤에 있으므로 어느 단계로 떨어져도 화면 코드는 동일하다.

- 어댑터: `MapAdapter` 인터페이스 (`app/src/map/MapAdapter.ts`)
- 로드 타임아웃 **5초** → 무한 로딩 금지
- **마커 헬스체크**: SDK 로드 성공 후 마커 1개를 실제로 생성해보고 실패하면 폴백. "기본 지도만 뜨는 함정" 대응.
- 각 어댑터는 **동적 import** → 리스트 폴백 시 Kakao/Leaflet 코드가 번들에 로드되지 않음

## 5. 외부 도메인 화이트리스트 (콘솔 등록 대상)

| 도메인 | 용도 | 단계 |
|---|---|---|
| `dapi.kakao.com` | Kakao Map JS SDK 로더 | 지도 ① |
| `t1.daumcdn.net` | **Kakao SDK가 동적 fetch하는 모듈 CDN** (geometry, marker 등) | 지도 ① |
| `*.tile.openstreetmap.org` | OSM 타일 | 지도 ② |
| `unpkg.com` 또는 self-host | Leaflet CSS/JS (self-host 권장 — 화이트리스트 항목 감소) | 지도 ② |
| 정적 JSON 호스트 (Vercel/CF Pages 도메인) | 시군구 JSON | 필수 |
| 프록시 도메인 (Vercel Functions) | TAGO·기상청·로깅 | 필수 |

**`t1.daumcdn.net` 누락이 커뮤니티 무한로딩 사례의 유력한 원인이다.** 지도 ①을 시도한다면 이 도메인을 반드시 함께 등록해야 한다.

## 6. 미해결 (사람 확인 필요)

- 실기기에서 Kakao SDK 동작 여부 → 폴백 체인이 자동 처리하므로 **블로커 아님**
- 콘솔 도메인 화이트리스트 등록 절차·승인 소요
- 개인(비사업자) 입점 가능 여부, 콘솔 심사 소요 기간 ← **일정 최대 블로커**
- 공공데이터포털 운영계정 활용신청 승인
