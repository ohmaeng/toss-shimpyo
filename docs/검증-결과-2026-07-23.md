# 실 API 검증 결과 (2026-07-23)

실제 공공데이터포털 키로 프록시·파이프라인 엔드포인트를 전부 호출해 검증했다.
아래는 "문서상 미확인"이었던 것들을 실제 응답으로 확정한 기록이다.

---

## 1. 프록시 — 3개 버그 수정 후 완전 작동 ✅

배포: `https://toss-shimpyo-proxy.vercel.app` (Vercel, `runtime: 'edge'`)

| # | 버그 | 증상 | 수정 |
|---|---|---|---|
| 1 | 함수가 Web 표준 시그니처인데 `runtime:'nodejs'` | 500 `FUNCTION_INVOCATION_FAILED` | → `runtime:'edge'` |
| 2 | 기상청에 `_type=json` 전송 (기상청은 `dataType=JSON`) | XML 반환 → `res.json()` 실패 → 503 | → `dataType=JSON` |
| 3 | 도착정보 오퍼레이션명 오타 `getSttnAcctoArvlPrarngInfoList` | 404 `API not found` | → `getSttnAcctoArvlPrearngeInfoList` |

검증 응답:
- `GET /api/weather?lat=36.3504&lng=127.3845` → `{"kind":"ok","tempC":30.1,"heatAlert":null}`
- `GET /api/arrivals?cityCode=25&nodeId=DJB8001378` → `{"kind":"ok","items":[]}` (그 시각 도착 예정 버스 없음 = 정상)

**키·활용신청은 처음부터 정상이었다.** 원인은 전부 코드 버그였다.

## 2. 앱 — CORS 버그 수정 ✅

`app/src/platform/analytics.ts`의 `/api/log` 전송이 `navigator.sendBeacon`을 썼는데,
sendBeacon은 credentials를 강제 포함한다. 프록시가 `Access-Control-Allow-Origin: *`(와일드카드)를
주므로 "wildcard + credentials" 조합이 브라우저에서 차단됐다(로깅 유실).

→ `fetch(keepalive:true, credentials:'omit')` 로 교체. credentials 없이 전송되어 CORS-safe.
arrivals/weather는 `http.ts`에서 credentials 없이 fetch하므로 애초에 영향 없었다.

## 3. 파이프라인 오퍼레이션명 — 전부 정상 ✅

arrivals에서 겪은 오타가 파이프라인에도 있는지 실호출로 확인 → **3개 모두 정상**:

| 상수 | 오퍼레이션 | 결과 |
|---|---|---|
| `STOP_INFO_URL` | `BusSttnInfoInqireService/getSttnNoList` | 200, gpslati/gpslong/nodeid 반환 ✅ |
| `ROUTE_LIST_URL` | `BusRouteInfoInqireService/getRouteNoList` | 200, routeid/routeno 반환 ✅ |
| `ROUTE_INFO_URL` | `BusRouteInfoInqireService/getRouteAcctoThrghSttnList` | 200, 경유정류장 반환 ✅ |

TAGO는 `_type=json`을 쓴다(기상청과 반대) — 파이프라인 코드가 이미 맞게 쓰고 있다.

## 4. ⚠️ 배차간격 확보율 = **0%** — 사람 결정 필요

**이게 이번 검증의 가장 중요한 발견이다.**

`getRouteNoList` 응답 필드는 4개 도시(인천 23·부산 21·대전 25) 전부 동일하게:
```
endnodenm, endvehicletime, routeid, routeno, routetp, startnodenm, startvehicletime
```
**첫차·막차 시각만 있고 배차간격(intervaltime) 필드가 아예 없다.** 파이프라인의
`_parse_interval` 후보 필드(`intervaltime`/`intervalsattime`/`intervalsuntime`)는 0/100 존재.

→ **TAGO는 배차간격을 제공하지 않는다. 확보율은 구조적으로 0%다.**

### 이것이 발동시키는 컷 규칙 (README 불변규칙 #4, 개발계획서 §7.3)
> 배차간격 확보율이 20% 미만이면 대안 정류장 제안(4중 조건 #4)은 화면에 거의 뜨지 않는다.
> 그 경우 대안 정류장 + 데이터셋 ⑤를 M4로 이월하고, 차별점을 다른 축에서 찾는다.

0% 이므로 **대안 정류장 기능은 현재 유령 기능**이다(코드는 있으나 절대 안 뜸).

### 결정이 필요한 것 (AI가 못 정함 — `docs/사람이-해야-할-일.md` 참조)
1. **대안 정류장 기능을 뺄 것인가, 다른 배차간격 소스를 찾을 것인가?**
   - TAGO 외 소스: 지자체별 BIS, 국가대중교통DB(TAGO와 별개), 서울 TOPIS 등. 통일된 전국 소스는 없다.
   - 빼면 차별점("기다리는 동안 시원한 곳")을 **쉼터 결합**에만 걸어야 한다.
2. 차별점을 쉼터 쪽으로 옮긴다면, 무더위쉼터 데이터 품질(운영시간 컬럼 유무)이 새 관건이 된다.

## 5. ⚠️ TAGO cityCode 정정 필요

- **춘천 = `32010`** (강원). 로드맵의 `--city-codes 23,25,51` 중 **51은 TAGO에 존재하지 않는다**(0건).
  - 앱/목데이터의 `51110`은 행정 법정동 코드이지 TAGO cityCode가 아니다.
- **서울은 TAGO 도시코드 목록에 없다.** → 강남 예시 정류소는 실시간 도착정보를 받을 수 없다(서울은 별도 API).
  - 실데이터 데모 도시는 **인천(23)·대전(25)·춘천(32010)** 처럼 TAGO 커버 지역이어야 한다.

올바른 실데이터 빌드 예:
```bash
python build_data.py --city-codes 23,25,32010 --shelter-csv ./shelters.csv --data-date 2026-07
```
