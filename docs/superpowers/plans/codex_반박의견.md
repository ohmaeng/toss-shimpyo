## 1. 총평

현재 기획은 데모와 스토리텔링은 강하지만, 본선 배점이 가장 큰 **데이터 처리구조·분석 타당성 20점**에서 정면 공격받을 치명적인 산식 오류가 있다. 이 상태로는 대상보다 “잘 만든 서비스 시연”에 가깝게 평가될 위험이 크다.

HWP에 명시된 본선 평가 기준은 다음과 같다.

| 심사항목 | 세부 항목 | 배점 |
|---|---|---:|
| 정책 실효성 | 지역 현안 적합성, 실제 행정 활용 가능성 | 15 |
| 정책 실효성 | 시민 체감 효과, 확장 가능성 | 15 |
| 기술 구현력 | 데이터 처리구조, 분석의 타당성 | 20 |
| 기술 구현력 | 기능 구현 수준, 시연 가능성 | 20 |
| 서비스성 | 사용자 편의성, 전달력, 화면구성, 활용 시나리오 | 20 |
| 데이터 활용성 | 춘천 관련 공공데이터 활용 적절성 | 5 |
| 데이터 활용성 | 데이터 결합·분석 구조의 타당성 | 5 |

아래 쟁점은 예상 감점 위험이 큰 순서로 정리했다.

## 2. 치명적 문제

### 2.1 “미확인 ≠ 없음” 원칙과 실제 산식이 모순된다

설계는 미확인을 없음으로 취급하지 않는다고 선언하지만, 실제 산식은 다음과 같다.

- `comfortScore`: `yes`만 가점하고 `no`와 `unknown`은 모두 0
- `U = 1 - comfortScore`
- 따라서 `unknown`과 `no`가 쉼표지수에서 같은 결핍 효과를 갖는다.

그늘 정보가 미확인인 1,886개 정류장이 모두 “그늘이 없는 것과 같은 효과”로 투자 우선순위에 반영된다. 이를 “쉼 미확인도”라고 이름 붙여도 순위에 미치는 수치 효과가 같으므로 방어가 어렵다.

이는 설계가 정한 다음 원칙을 스스로 위반한다.

> 미확인을 ‘없음’으로 취급하는 산식 금지

심사위원 예상 질문:

> “정보가 없는 정류장을 왜 시설이 부족한 정류장처럼 높은 순위에 올렸습니까?”

#### 권고 수정

시설 항을 다음처럼 분리한다.

- `confirmedDeficit`: 현장·로드뷰에서 `no`로 확인된 시설만 결핍으로 계산
- `unknownRate` 또는 `dataConfidence`: 별도 표시하고 조사 우선순위에만 사용
- 개선 우선순위에는 확인된 `yes/no` 시설만 반영
- 확인된 시설 정보가 부족하면 해당 항을 결측 처리하거나 신뢰도 배지를 표시

현재 시설 `no`가 0건이라면 “현재 데이터만으로 시설 결핍 기반 설치 순위는 산출할 수 없다”고 인정하는 편이 분석적으로 정직하다. 대신 다음 2단계 의사결정 구조가 타당하다.

1. 수요 기반 **조사 우선순위** 산출
2. 조사로 `no`가 확인된 후 **시설별 설치 검토 우선순위** 산출

### 2.2 B2C “편안 경로”도 미확인 정류장을 불편한 정류장으로 취급한다

시민 경로 정렬에서도 `unknown=0`, `no=0`이다. 시설 정보가 단지 미확인인 가까운 정류장이 확인된 정류장보다 뒤로 밀린다.

따라서 현재 알고리즘은 “편안 경로”가 아니라 **확인된 편의시설 우선 경로**에 가깝다. 특히 그늘 확인이 4곳뿐이므로 춘천 전역에서 실제로 추천 차이가 발생하는 검색 비율이 매우 낮을 수 있다.

#### 권고 수정

- “편안 경로”를 “편의시설 확인 경로”로 정직하게 표현
- 미확인 후보를 결핍으로 감점하지 않고 정보 신뢰도를 별도 표시
- 사용자가 “확인된 편의시설 우선”과 “최단 도보”를 직접 선택
- 다음 실제 유효성 지표를 측정
  - 전체 검색 중 대안 후보가 2개 이상인 비율
  - 편의시설 확인 후보가 존재하는 검색 비율
  - 추천으로 늘어난 평균 도보시간
  - 확인 시설을 얻기 위해 추가되는 도보시간 분포

현재 구현 계획에는 추천 알고리즘의 실효 커버리지 검증이 없다.

### 2.3 “수혜 승차 건수”는 인과적 과장이다

예산 시뮬레이터는 선정 정류장의 기존 승차량을 더해 “수혜 승차 건수”로 표시한다. 그러나 기존 승차량 합계는 다음을 뜻하지 않는다.

- 시설 설치로 실제 혜택을 받는 고유 시민 수
- 일일 이용자 수
- 폭염 시간에 새 시설을 실제 이용할 사람 수
- 중복 제거된 수혜자 수

동일인이 여러 번 승차할 수 있고, 원본은 4일 표본이며 정류장명 기준 양방향 합산이다. 원본 집계 단위를 검증하지 않은 채 `dailyBoardings`라고 부르는 것도 위험하다.

#### 권고 수정

- “수혜 승차 건수” → “대상 정류장의 표본기간 한낮 승차 합계”
- 공식 일평균 환산 근거가 있을 때만 “추정 일평균 승차” 사용
- “수혜자” 표현은 사용하지 않거나 별도 추정모형과 가정을 공개
- 표본 기간, 양방향 합산, 결측 제외 수를 결과 옆에 항상 표시

### 2.4 투자 대상과 예산 단위가 일치하지 않는다

우선순위 지수는 의자·그늘·조명·수요·노선·POI를 합치지만 예산 시뮬레이터는 “벤치 1기 단가” 하나로 모든 상위 정류장을 커버한다.

현재 구조로는 다음을 구분하지 못한다.

- 벤치가 필요한 곳
- 그늘막이 필요한 곳
- 조명이 필요한 곳
- 단순히 정보가 미확인인 곳
- 이미 벤치가 있지만 다른 항목 때문에 높은 점수를 받은 곳

행정 적용에서는 “상위 10곳”보다 **어떤 시설을 왜 설치할 것인가**가 중요하다.

#### 권고 수정

정책 단위로 순위와 예산을 분리한다.

- 벤치 설치 후보
- 그늘막 설치 후보
- 조명 보강 후보
- 현장조사 후보

각 트랙에 시설별 단가, 설치 가능 조건, 기존 시설 중복 여부를 적용해야 한다.

## 3. 분석 타당성의 주요 약점

### 3.1 기본 동일 가중치 25:25:25:25에는 근거가 없다

가중치 슬라이더는 투명성을 높일 뿐 타당성을 자동으로 만들지 않는다. “가중치는 담당자가 정한다”는 설명은 판단 책임을 사용자에게 넘기는 것으로 보일 수 있다.

수요, 시설, 노선 수, POI는 측정 의미와 데이터 품질이 모두 다르다. 이들을 같은 가중치로 시작하는 근거가 필요하다.

#### 권고 수정

- 최소 3개 정책 프리셋 제공
  - 폭염 대응형
  - 고령자 이동지원형
  - 이용량 중심형
- 가중치 변화에 따른 Top 10 진입 빈도 계산
- 순위 상관계수 또는 순위 변동 폭 표시
- “상위권이 유지되면 강건하다”를 육안이 아니라 표나 그래프로 검증

### 3.2 노선 수는 수요와 중복될 수 있고 의미가 불명확하다

노선 수를 “환승·대기 발생량 프록시”로 사용하지만 검증 근거가 없다.

- 노선 수가 많다고 대기시간이 긴 것은 아니다.
- 노선 수가 많다고 환승이 많은 것도 아니다.
- 승차량과 상관이 높다면 동일 현상을 이중 가중한다.
- 2025년 11월 노선 조정 전 수요와 2026년 노선 정보를 함께 쓰면 시점이 불일치한다.

#### 권고 수정

- 수요와 노선 수의 상관분석 수행
- 노선 개편 전후 정합성 확인
- 근거가 약하면 노선 수는 지수에서 제외하고 설명용 원값으로만 표시

### 3.3 POI 취약도 정의가 지나치게 거칠다

현재 계획은 병원·경로당·시장 300m 안의 “종류 수 ÷ 3”을 취약 가중치로 사용한다.

- 병원 10개와 병원 1개가 동일하다.
- 종합병원과 소형 의원이 동일하다.
- 경로당 규모와 이용인원이 반영되지 않는다.
- 직선거리 300m는 경사, 횡단보도, 하천·철도 등 보행 장벽을 무시한다.
- 실제 고령인구 또는 교통약자 이용 데이터가 없다.
- 300m 기준의 근거가 없다.

“취약 가중치”라는 명칭은 사람의 취약성을 측정하는 것처럼 들리지만 실제로는 POI 인접도다.

#### 권고 수정

- 명칭을 “생활지원시설 인접도”로 변경
- 고령인구 지표와 POI 지표를 구분
- 300m 기준의 정책·연구 근거 제시
- 가능하면 보행 네트워크 거리 사용
- 데이터가 확보되지 않으면 P0 지수에서 제외

### 3.4 분위수 정규화는 데이터 품질 문제를 해결하지 않는다

사용 데이터는 4일 표본, 정류장명 양방향 합산, 88.2% 커버리지, 노선 개편 전 자료다. 분위수 변환은 극단값 영향을 줄일 뿐 표본 편향과 조인 오류를 없애지 않는다.

현재 계획에 다음 검증이 빠져 있다.

- 평일·주말 포함 여부
- 우천·폭염 등 표본 4일의 특이성
- 동명 정류장 오결합 사례
- 양방향 합산이 정책 단위를 왜곡하는 사례
- 결측 223곳의 공간적·지역적 편향
- 하루씩 제외했을 때 Top 20 안정성
- 상위 정류장 원본 대조 샘플

대상 경쟁에서는 복잡한 산식보다 이 검증 결과가 더 중요한 증거가 될 수 있다.

### 3.5 시설 데이터 신뢰도가 낮은데 지수의 핵심 축처럼 보인다

현재 그늘 데이터는 `yes=4`, `no=0`, `unknown=1886`이다. 따라서 “춘천 정류장 중 그늘이 4곳뿐”이라고 말할 수 없다. 정확한 표현은 다음과 같다.

> “공공데이터 결합으로 정류장 인접 그늘막 4곳을 확인했으며, 1,886곳은 없음이 아니라 미확인입니다.”

“그늘 4곳/1,890 정류장”을 문제 제기 첫 문장으로 단독 사용하면 데이터 과장으로 공격받을 수 있다.

## 4. 정책 실효성의 약점

### 4.1 행정 채택 구조가 화면과 출력물에서 끝난다

CSV와 A4 출력은 채택 가능성을 보조하지만 실제 행정 프로세스가 빠져 있다.

누락된 질문은 다음과 같다.

- 어느 부서가 데이터를 소유하고 갱신하는가?
- 조사 목록을 누가 현장조사 과업으로 전환하는가?
- 시설 설치 가능 여부는 누가 검토하는가?
- 보행폭, 도로점용, 사유지 등 설치 불가 조건은 어떻게 반영하는가?
- 예산 수립 시 어떤 주기로 재계산하는가?
- 연간 운영·갱신 비용은 얼마인가?
- 선정 결과에 민원이 들어오면 어떻게 설명하고 수정하는가?

대상 수준이라면 다음 업무 흐름이 보여야 한다.

```text
데이터 미확인
  → 조사 배정
  → 사람 확인
  → 시설별 설치 후보
  → 현장 적합성 검토
  → 예산안 출력
  → 설치 후 데이터 갱신
```

### 4.2 실제 사용자 검증 계획이 없다

서비스성은 20점이지만 테스트 계획은 대부분 코드 테스트와 팀 내부 육안 확인이다. 고령자를 핵심 사용자로 설정했다면 실제 사용자 검증이 필요하다.

최소 권고 검증:

- 60대 이상 사용자 3~5명 태스크 테스트
- QR 등록 성공률
- 목적지 검색 완료율
- 설명 없이 완료하는 데 걸린 시간
- 잘못 누른 횟수
- “편안 경로” 의미를 제대로 이해한 비율
- 글자 크기와 버튼 크기 만족도

폰트 18px와 터치타깃 48px만으로 사용자 편의성이 증명되지는 않는다.

### 4.3 WCAG AA 선언과 키보드 접근성 기준이 어긋난다

계획은 WCAG AA를 선언하면서 “키보드 없이 핵심 흐름 완결”을 접근성 하한으로 둔다. 웹 접근성은 오히려 **키보드만으로도** 핵심 흐름을 완료할 수 있어야 한다.

필요한 검증:

- 키보드 전용 조작
- 포커스 순서와 포커스 표시
- 스크린리더 레이블
- 색상 외 상태 표현
- 지도 없이 동일 정보 접근
- 슬라이더의 키보드·스크린리더 대체 입력
- `prefers-reduced-motion`
- 실제 명암비 측정

## 5. 구현·시연 리스크

### 5.1 오프라인 데모의 범위가 정의되지 않았다

“와이파이 차단 후에도 흐름 계속 동작”은 다음 조건에 따라 결과가 달라진다.

- 앱을 이전에 한 번 열어 캐시했는가?
- 목적지·노선 데이터가 서비스워커에 캐시됐는가?
- 지도 타일이 필요한가?
- OSRM 요청 전인가 후인가?
- 새 QR URL을 오프라인 상태에서 처음 여는가?
- 브라우저 사용인가 설치된 PWA 사용인가?

QR 링크를 처음 여는 순간 인터넷이 없으면 정적 앱 자체에 접근하지 못할 수 있다.

#### 권고 수정

- “최초 1회 접속 후 오프라인 사용 가능”처럼 조건을 정확히 표현
- 오프라인 E2E 매트릭스 추가
- 지도 타일 없는 상태의 문장·목록 폴백 검증
- OSRM 실패 전환 시간과 UI 문구 검증
- 새로고침, 직접 URL 진입, QR 진입을 각각 테스트

### 5.2 P0 범위가 과도하다

Phase 4까지 P0로 잡으면 사실상 다음을 모두 필수로 구현해야 한다.

- 핵심 산식
- B2C 경로 추천
- QR
- 대시보드와 지도
- 가중치·시간대 UI
- 근거 카드와 CSV
- 예산 시뮬레이터
- 150곳 로드뷰 조사
- AI 판독 파이프라인
- POI 3종 확보·지오코딩·결합
- 배포·인쇄·오프라인 리허설

HWP 일정상 2일차 08:30부터 제출자료 취합이 시작된다. OT, 식사, 멘토링, 중간점검을 제외하면 실제 집중 개발시간은 계획보다 짧다.

#### 권고 P0 축소

1. 검증된 조사/설치 2단계 우선순위
2. 시민 편의시설 확인 경로
3. 행정 근거 카드와 시설별 예산안

AI 브리핑과 익명 카운터는 대상에 필수적인 기능이 아니며 설명·보안 부담을 늘린다.

### 5.3 AI 로드뷰 “조사 비용 1/5”는 근거 없는 주장이다

사람이 로드뷰 열기, 방향 찾기, 캡처, AI 결과 검수, 판별불가 재확인을 모두 수행한다. 사람이 직접 판정하는 것보다 정말 5배 빠른지 측정되지 않았다.

#### 권고 검증

최소 20~30곳에 대해 비교한다.

- 사람 단독 평균 처리시간
- AI 보조 평균 처리시간
- 시설별 정밀도·재현율
- 판별불가율
- 사람 수정률

검증 전에는 다음 수준으로만 주장한다.

> “AI 보조로 로드뷰 판독 결과의 초안 작성과 문서화를 표준화했습니다.”

### 5.4 익명 카운터의 개인정보 방어가 완전하지 않다

클라이언트 페이로드에 IP가 없다는 것과 서버가 IP를 처리·기록하지 않는 것은 다르다. Supabase, Cloudflare, 프록시 또는 호스팅 로그에서 IP가 자동 처리될 수 있다.

추가로 불명확한 항목:

- 옵트인 UI와 동의 철회
- 보존 기간
- 중복·봇 이벤트 방지
- 반복 클릭 데이터의 의미
- QR 스캔과 즐겨찾기 추가 이벤트의 중복
- 모의 데이터와 실데이터의 격리

P1이라도 넣는 순간 보안과 정책 설명 부담이 크게 늘어난다. 삭제하거나 완전히 별도 실험 기능으로 격리하는 편이 안전하다.

## 6. 문서 및 계획 간 오류·불일치

### 6.1 `comfortScore` 수식의 괄호가 불명확하다

현재 표기는 분자 전체를 활성 가중치 합으로 나누려는 의도지만, 코드처럼 읽으면 마지막 항에만 나눗셈이 적용될 수 있다.

다음처럼 명시해야 한다.

```text
comfortScore =
  (seatScore + shadeScore + optionalLightScore)
  / activeWeightSum
```

### 6.2 설계의 “시간대 슬라이더”와 계획의 “3종 프리셋”이 다르다

설계는 시간대 슬라이더라고 하지만 구현 계획은 아침·한낮·저녁 프리셋으로 제한한다. 임의 시간창보다 프리셋이 표본 노이즈를 줄이는 데 유리하므로 설계를 프리셋으로 통일하는 편이 적절하다.

### 6.3 개선 트랙의 수요 결측 처리 원칙이 서로 다르다

설계는 개선 검토 우선순위를 수요 실측이 있는 정류장 대상으로 정의한다. 그러나 구현 계획은 수요가 없는 정류장도 나머지 항목으로 재정규화해 `index != null`인 순위를 생성한다.

권고안:

- 수요 결측 정류장은 설치 개선 순위에서 제외
- 조사 우선순위로 이동
- 또는 별도 “수요 미확인 개선 후보”로 분리하되 같은 순위표에 섞지 않음

### 6.4 역할 분담과 시간 예산이 없다

계획에는 역할 이름은 있지만 담당자, 예상 시간, 마감 시각, 선후행 관계, 실패 시 컷라인이 없다.

각 태스크에 다음을 추가해야 한다.

- 담당자
- 예상 소요시간
- 완료 마감 시각
- 선행 데이터·태스크
- 실패 시 제거할 기능
- 발표자료 담당
- 배포·실기기 검증 담당
- 발표자와 데모 조작자

## 8. 대상 가능성을 높이는 권고 구조

현재의 “쉼표지수 하나로 시민과 행정을 연결”하는 구조보다 다음 구조가 분석적으로 강하고 방어 가능하다.

### 8.1 시민용: 편의시설 확인 경로

- 확인된 시설이 있는 정류장을 우선 제시
- 미확인은 정보 부족으로 명시
- 최단 도보와 편의시설 확인 경로를 사용자가 선택
- 추천이 실제로 가능한 검색 비율을 공개

### 8.2 행정용 1단계: 조사 우선순위

- 수요가 높고 시설 정보가 부족한 곳을 조사 대상으로 선정
- 현재 데이터 상태만으로도 정당화 가능
- 결과는 조사 과업 목록과 CSV로 출력

### 8.3 행정용 2단계: 시설별 설치 검토 우선순위

- 조사에서 `없음`이 확인된 정류장만 대상
- 벤치·그늘막·조명별 순위를 별도 산출
- 수요, 생활지원시설 인접도, 설치 가능 조건을 시설별로 적용

### 8.4 시설별 예산 시뮬레이터

- 시설별 단가와 기준일 공개
- 이미 설치된 시설 제외
- “수혜자 수” 대신 대상 정류장 표본 승차량 표시
- 설치 가능성 검토 전후 결과를 구분

### 8.5 타당성 증명

- 원본 데이터 대조
- 결측 편향 분석
- 민감도 분석
- 실제 사용자 테스트
- AI 판독 정확도와 시간 절감 측정
- 오프라인 실기기 테스트

이 구조의 핵심 장점은 “미확인은 미확인으로”라는 프로젝트 정체성이 문구가 아니라 산식과 행정 흐름 전체에서 실제로 지켜진다는 점이다.

## 9. 우선순위별 수정 권고

### P0 — 구현 착수 전에 반드시 수정

1. `U = 1 - comfortScore` 폐기
2. 확인된 결핍과 미확인을 별도 변수·트랙으로 분리
3. 조사 우선순위와 설치 우선순위를 2단계로 재설계
4. 시설별 설치 순위와 시설별 예산 모델로 변경
5. “수혜 승차 건수” 표현과 `dailyBoardings` 정의 수정
6. 수요 결측 정류장의 순위 포함 규칙 통일
7. 데이터 품질 검증과 민감도 분석을 구현 계획에 추가

### P1 — 대상 경쟁력을 위해 필수

1. 고령 사용자 3~5명 태스크 테스트
2. 편의시설 확인 경로의 실효 커버리지 측정
3. 정책 프리셋과 순위 안정성 시각화
4. 시설별 단가 근거와 설치 가능 조건 추가
5. 행정 업무 흐름과 운영 주체 명시
6. 오프라인 E2E 테스트 매트릭스 추가

### P2 — 시간이 남을 때만

1. AI 로드뷰 판독 보조
2. AI 브리핑
3. 익명 시민 신호 카운터

AI와 카운터보다 분석의 정직성, 실제 사용자 검증, 행정 실행 구조가 본선 배점과 더 직접적으로 연결된다.

## 11. 최종 판단

현재 기획의 강점은 분명하다.

- 춘천시 실제 데이터 6종을 결합했다.
- 시민용과 행정용을 하나의 서비스로 연결했다.
- 시설 3상태와 출처 표기로 데이터 정직성을 강조했다.
- 오프라인, 큰 글씨, 가입 없는 QR 등 시연 포인트가 좋다.
- 근거 카드, CSV, 인쇄물은 행정 활용 서사에 유리하다.

그러나 핵심 산식이 “미확인 ≠ 없음” 원칙을 실제로 위반하면 이 모든 강점이 오히려 역공의 근거가 된다. 대상 전략의 최우선 과제는 기능을 더 추가하는 것이 아니라 다음 세 가지를 바로잡는 것이다.

1. **결핍과 미확인의 완전한 분리**
2. **조사와 설치의 2단계 정책 흐름**
3. **복합 점수보다 데이터·사용자·행정 검증 강화**

이 세 가지가 해결되면 “미확인은 미확인으로”라는 문장이 단순한 홍보 문구가 아니라, 분석 타당성과 행정 신뢰성을 동시에 증명하는 대상급 차별점이 될 수 있다.
