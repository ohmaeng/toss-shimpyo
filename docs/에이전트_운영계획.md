## 2. 페이즈별 실행 순서

### Phase 0 (~7/12) — 플랫폼 검증
```
ait-platform-scout (단독, 순차)
  ├─ 스파이크 ①~⑥ 실행
  └─ docs/확정-아키텍처-메모.md 산출
       ↓
map-adapter-engineer  ← 스파이크 ① 결과로 지도 단계 확정
       ↓
red-team-critic + scope-cut-marshal (병렬)
  → "스파이크 결과가 일정에 무슨 뜻인가"
```
**게이트**: 확정 아키텍처 메모 없이 M1으로 넘어가지 않는다.

### 데이터 파이프라인 (~7/14, M1과 병행)
```
data-pipeline-engineer (단독)
  → 조인 성공률·배차간격 확보율 실측 리포트
       ↓
red-team-critic  ← "대안 정류장 4중 조건을 만족하는 정류장 쌍이 실제로 몇 건인가"
```
이 숫자가 0에 가까우면 대안 제안 기능은 **존재하지 않는 기능**이다. 7/21 컷을 앞당길 근거가 된다.

### M1 (~7/18) — 코어 플로우
```
[병렬] frontend-engineer  |  api-proxy-engineer  |  toss-ui-designer
                    ↓
              three-state-auditor   (문구·기본값 감사)
                    ↓
[병렬] edge-case-qa  |  perf-budget-guard  |  retention-analyst(로깅 확정)
                    ↓
[병렬 관점 패널] persona-commuter | persona-elderly | persona-caregiver
                    ↓
              judge-reviewer  →  scope-cut-marshal
```
**M1 합격 기준**: 지도 없이도 코어 플로우가 성립한다. 이게 안 되면 M2로 못 간다.

### M2 (~7/25) — 지도 + 부가 기능
```
[병렬] map-adapter-engineer | frontend-engineer | toss-ui-designer
                    ↓
              three-state-auditor  (대안 정류장 판정 로직 집중)
                    ↓
[병렬] perf-budget-guard(마커 렌더) | persona 3종 | judge-reviewer
                    ↓
              red-team-critic
```
**7/21**: `scope-cut-marshal` 강제 실행. 컷 발동 시 대안 정류장 + 데이터셋 ⑤를 M4로 이월.

### M3 (7/26~29) — 제출
```
compliance-checker (필수 · P0 위반 시 제출 차단)
        ↓
edge-case-qa (전체 재실행)  +  three-state-auditor (전체 재실행)
        ↓
judge-reviewer (최종)  →  사람이 콘솔 제출
```

### M4 (8/1~26) — 심사 기간 운영
```
매주:  retention-analyst (지표 판독 → 개선 우선순위)
        ↓
      data-pipeline-engineer (주 1회 재실행 · dataDate 갱신)
        ↓
      frontend-engineer (개선 배포)
        ↓
      three-state-auditor + edge-case-qa (회귀 검사)
```

## 3. 병렬 조합 (한 번에 띄우면 좋은 세트)

**관점 패널** — UI가 바뀔 때마다 4개를 동시에:
`persona-commuter` + `persona-elderly` + `persona-caregiver` + `judge-reviewer`
서로 다른 렌즈라 중복이 적고, 충돌하는 지적이 나오는 지점이 진짜 설계 문제다.
(예: 직장인은 "탭을 줄여라", 고령자는 "버튼을 키워라" → 화면 밀도 충돌)

**게이트 패널** — 마일스톤 종료 시 3개 동시에:
`three-state-auditor` + `perf-budget-guard` + `edge-case-qa`

**전략 패널** — 큰 결정 앞에서 2개 동시에:
`red-team-critic` + `scope-cut-marshal`

## 4. 권한 구조 (누가 무엇을 이길 수 있는가)

| 상황 | 결정권 |
|---|---|
| [불변 규칙] 위반 (3상태·폴백 순서·4중 조건·컷 규칙) | 아무도 못 뒤집는다. 사람만 명시적으로 변경 가능 |
| 규정 위반 (P0) | `compliance-checker` — 제출 차단권 |
| 일정 충돌 | `scope-cut-marshal` — 기능 컷 결정 |
| 성능 예산 초과 | `perf-budget-guard`는 숫자와 선택지만 제시, 컷은 marshal |
| 페르소나끼리 충돌 | 사람이 결정. 에이전트가 합의하려 들지 않는다 |

**중요**: 관점 에이전트(페르소나·심사위원·레드팀)는 **코드를 고치지 않는다.** 지적만 하고, 수정은 빌드 에이전트가 한다. 관점과 구현을 같은 에이전트에 맡기면 자기 코드를 변호하기 시작한다.

## 6. 안 만든 에이전트와 그 이유

- **접근성 전담 에이전트**: `persona-elderly`(가독성·터치)와 `toss-ui-designer`(대비비·색각)가 나눠 가졌다. 별도로 두면 셋이 같은 말을 한다.
- **테스트 작성 에이전트**: 3주 프로젝트다. `edge-case-qa`가 실제 재현으로 검증하는 편이 유닛 테스트 커버리지보다 심사에 직결된다.
- **문서 작성 에이전트**: 산출 문서는 3개뿐이다(확정 아키텍처 메모, 화이트리스트, 이 문서). 각 담당이 직접 쓴다.
- **보안 전담**: `compliance-checker`에 키 노출·개인정보 항목을 넣었다. 이 앱은 백엔드가 거의 없다(프록시 1개).
