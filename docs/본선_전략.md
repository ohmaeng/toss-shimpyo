# 쉼표 정류장 — 본선 경쟁 전략

> 파생 문서. 제품 정의는 `제품_기획.md`, 데이터·AI 세부 계약은 `데이터_AI_계약.md`를 따른다.
> 외부 기준: 춘천시·강원도민일보 공고의 `데이터베이스 활용 시민 체감형 서비스 구현`, 지정과제 `생활 접근성 개선 서비스`. 경쟁 비교 원본: `C:\Users\user\Downloads\해커톤 본선진출(16팀).hwp`의 제목 목록.

## 기억시킬 문장

**버스가 올 때까지 기다리는 시민에게 도착정보와 대기환경을 안내하고, 그 자리에서 말한 불편을 AI가 정류장 단위의 현장확인 과업으로 전환합니다.**

## 대상 가능성 평가

| 심사축 | 현재 가능성 | 증명해야 할 것 |
|---|---|---|
| 공공문제 적합성 | 높음 | 시설정보 결측·고장·불편이 실제 행정 과업으로 연결되는 장면 |
| 공공데이터 활용 | 높음 | 춘천시·국토부·TAGO·BIT 자료의 출처·시점·충돌을 분리한 근거 카드 |
| AI 필연성 | 중상 | 자유발화 구조화, 중복 묶기, 공식자료 충돌 설명의 정량 평가 |
| 시민 체감 | 높음 | 고령 이용자가 QR부터 제보 완료까지 실제로 수행하는 테스트 |
| 차별성 | 중상 | 일반 민원앱이 못 하는 정류장 ID 연결과 시민 화면 환류 |
| 실행 가능성 | 중간 | 정류장 식별 오류, 데이터 갱신권한, 공식 민원 미연계를 정직하게 제한 |
| 확장성 | 높음 | 폭염·한파·야간·BIT 유지보수를 같은 데이터 계약으로 확장 |

현재 상태에서 대상 가능성은 **충분하지만 조건부**다. 화면 수보다 아래 한 사이클의 완결성과 AI 평가 결과가 당락을 좌우한다.

## 차별성

| 비교 대상 | 기존 방식 | 쉼표 정류장 |
|---|---|---|
| 지도·교통 앱 | 도착·경로 중심 | 도착정보와 방향별 대기환경, 불편 환류 결합 |
| 일반 민원 앱 | 시민이 위치·대상·내용을 직접 작성 | 현재 정류장 식별, 음성 구조화, 공식 시설자료 자동 대조 |
| 시설 현황 대시보드 | 정적 공공데이터 조회 | 시민신호·시점 충돌을 찾아 현장확인 과업 생성 |
| AI 챗봇 | 답변·요약 중심 | 정류장 ID와 시설 관측이력에 연결된 검증 가능한 행정 워크플로 |

차별화의 핵심은 `음성`, `QR`, `대시보드` 각각이 아니라 다음 폐쇄형 순환이다.

```text
시민 편익 → 현장신호 → AI 구조화·대조 → 사람 확인 → 데이터 갱신 → 시민 편익 향상
```

## 본선 16팀 제목 기준 경쟁 구도

제공된 `해커톤 본선진출(16팀).hwp`의 사업명·참가과제 제목만 비교했다. 다른 팀의 세부 기획서와 구현은 보지 못했으므로 우열이 아니라 **겹칠 가능성**을 판단한다.

| 경쟁군 | 확인된 제목 예시 | 쉼표 정류장의 방어선 |
|---|---|---|
| 교육·학생 매칭 | 봄내틔움, AI 교육자원 매칭 플랫폼, 에듀브릿지, 학생 교육자원 접근성 지도 | 대상·데이터·업무가 달라 직접 중복 낮음 |
| 지역경제·예약 | 춘천골목대장, 춘천 시내 로컬 예약 솔루션, 봄내마실, 일상의 춘천 | 직접 중복 낮음. 시민 체감성과 화면 완성도는 경쟁 가능 |
| 생활접근성 | 생활틈, 타봄 거점 재배치·반납 관리, 팜플렛 | 같은 지정과제라 가장 직접적인 평가 경쟁군 |
| 교통·도시행정 AI | 교통사고 저감 정책 제안, AI 도시 인프라 실시간 통합관제 | 공공데이터·AI·정책 실행성에서 직접 경쟁 |
| 정책 탐색 | 청년 정책 탐색 서비스 | 행정 연결을 주장할 수 있으나 쉼표 정류장은 현장 상태 환류로 구별 |

교통사고·통합관제 팀은 데이터 규모와 기술 인상이 강할 가능성이 있다. 쉼표 정류장은 더 큰 AI를 주장하기보다 `실제 시민 발화 1건이 어떤 공공자료와 충돌하고, 누가 확인하며, 시민 화면이 어떻게 바뀌는지`를 끝까지 증명해야 이긴다. 생활접근성 팀과는 고령자 사용성 실측과 방향별 정류장 데이터의 정밀도로 차별화한다.

## 가장 강한 라이브 데모

1. 공통 QR로 시민 앱에 진입해 첫 화면에서 `목적지로 가는 길 찾기`를 누른다.
2. 목적지를 말하고 인식 결과를 확인한다.
3. 앱이 현재 위치와 목적지로 올바른 승차 정류장을 자동 산출하고 버스·도착시간·대기환경을 보여준다.
4. 결과 화면에서 시민이 “버스 오는 화면이 며칠째 꺼져 있어요”라고 말한다.
5. AI가 `BIT 작동 불량·지속성 표현·산출된 승차 정류장`으로 구조화하고 사용자가 확인한다.
6. 관리자 웹에서 공식 BIT 설치자료와 대조되어 `설치 여부 문제가 아닌 고장 가능성`으로 분류된다.
7. 같은 문제의 반복 신고가 묶이고 담당자가 현장조사를 승인한다.
8. 조사 결과를 확정하면 시민 앱이 `BIT 점검 중/확인일`로 갱신된다.

## 공격 질문과 답변

| 질문 | 답변 |
|---|---|
| 기존 민원 앱과 무엇이 다른가? | 정류장 식별, 공공 시설자료 대조, 중복 묶음, 시민 안내 재반영이 한 흐름이다. |
| AI가 꼭 필요한가? | 버튼 집계가 아니라 시민 자유발화를 시설·문제·지속성으로 구조화하고 여러 시점의 자료 충돌을 설명한다. 정확도는 검증셋으로 제시한다. |
| AI가 틀리면? | 제출 전 시민 확인, 관리자 원문·근거 확인, 사람 확정 전 상태변경 금지의 세 단계가 있다. |
| 조회량이 실제 승차량인가? | 아니다. 승차예정 정류장 기반 서비스 신호이며 공식 승차자료와 분리한다. |
| 이용자가 고령자인가? | 알 수 없다. 고령자 친화 UX의 사용성은 시험하고, 연령별 수요는 공식 익명 집계만 사용한다. |
| 공통 QR이면 맞은편을 틀리지 않나? | 자동 확정하지 않고 4자리 번호·방면·랜드마크로 결과 단계에서 확인한다. |
| 제보가 많으면 설치하는가? | 아니다. 반복 제보는 현장확인 신호이고 공사·예산은 별도 행정결정이다. |
| 카카오맵이 최신이면 공식자료보다 우선인가? | 민간 지도는 교차검토 참고다. 공식 운영근거와 현장확인 결과를 분리해 보존한다. |

## 추가하면 경쟁력이 커지는 요소

### 반드시 유용

- AI 구조화 결과를 사용자가 제출 전에 수정하는 확인 화면
- 공식자료와 시민신호의 충돌 이유를 한 문장으로 설명하는 근거 카드
- `AI 권고/담당자 확정` 상태를 시각적으로 분리한 감사로그
- 고령 사용자 5명 이상 과업 성공률·완료시간·오류 유형 비교
- AI 검증셋과 시설/문제유형별 정확도, 실패 사례 공개
- 현장조사 과업 CSV/A4 출력과 결과 재입력

### 조건부 확장

- 공식 민원 시스템 연계: 춘천시 협의·인증·양식 API가 있을 때
- 연령별 수요: 교통카드 운영기관의 방향별 익명 집계 확보 시
- 목적지·경로 즐겨찾기: 핵심 길찾기와 행정 환류 완성 후 확장
- 사진 제보: 개인정보·얼굴·차량번호 마스킹과 저장정책 확보 시

### 넣지 말아야 할 장식

- 근거 없는 AI 공사순위
- 로그인으로 나이를 자동 확보한다는 설명
- 제보 건수를 수혜자 수로 변환한 지표
- 기능과 연결되지 않는 생성형 AI 채팅창
- 정확하지 않은 정류장 ID를 강제 매칭한 완성도 연출

## 본선 전과 현장 작업 구분

- 사전 허용: 기획 기준서, 데이터 목록·결측, 스키마, 와이어프레임, 스토리보드, 검증 계획, 발표 틀
- 현장 증명: 핵심 데이터 연결, AI 구조화·대조, 시민–행정 환류, 실제 분석 결과, 사용자·모델 평가
- 발표: 기존 코드 재사용, 사전 준비, 본선 신규 구현을 커밋과 작업표로 구분

## 최종 판정

대상을 노릴 수 있는 이유는 `노인용 버스 앱`이어서가 아니라 **시민이 얻는 즉시 편익을 공공데이터 품질과 행정 실행으로 되돌리는 AI 순환 구조** 때문이다. 반대로 AI 정량평가와 사람 확정 과정이 빠지면 일반 민원앱에 도착정보를 붙인 수준으로 평가될 가능성이 높다.
