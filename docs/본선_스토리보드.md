# 쉼표 정류장 — 본선 스토리보드

> 파생 문서. 시민은 모바일 앱, 관리자는 데스크톱 웹이다. 실제 장면·행동·시스템 반응을 담은 JPG는 `visuals/시민_UI_스토리보드.jpg`, `visuals/행정_AI_스토리보드.jpg`다.

## 시민 앱 스토리보드

| 장면 | 화면 | 사용자 행동 | 시스템 반응 |
|---:|---|---|---|
| 1 | QR 진입 | 공통 QR을 스캔 | 로그인 없이 시민 앱 첫 화면 실행 |
| 2 | 첫 화면 | `목적지로 가는 길 찾기` 또는 `정류장 불편 알리기` 선택 | 선택한 업무로 이동 |
| 3 | 목적지 말하기 | 목적지를 말하거나 문자 입력 | 음성을 텍스트로 변환 |
| 4 | 목적지 확인 | 인식 결과가 맞는지 확인 | 맞으면 경로 계산, 아니면 재입력 |
| 5 | 경로 계산 | 잠시 기다림 | 현재 위치와 목적지로 탈 버스·올바른 승차 정류장 자동 산출 |
| 6 | 한 화면 길 안내 결과 | 추천 버스·총 소요시간·버스 도착예정·타는 곳까지 도보시간·목적지 도착예상·시설 확인 | 탈 수 없는 버스 제외 후 목적지 도착예상시간 순으로 제공 |
| 7 | 불편 신고 시작 | 위치 또는 정류장명·4자리 번호를 말하거나 입력 | 대상 정류장 확인 후 신고 입력 제공 |
| 8 | AI 신고 확인 | 구조화 결과를 확인·수정하고 제출 | 검수 전 시민 현장신호로 저장 |

2번에서 3번과 7번으로 갈라진다. 6번의 `이 정류장에서 불편한 점 알리기`도 8번 흐름으로 연결한다. 주변 정류장 선택 전용 화면은 없다.

## 관리자 웹 스토리보드

| 장면 | 웹 화면 | 관리자 행동 | 시스템 반응 |
|---:|---|---|---|
| 1 | 대시보드 | 신규 신호·자료 충돌·현장확인 대기 확인 | 업무 상태별 건수와 목록 제공 |
| 2 | 신고 검토 | 시민 원문과 AI 구조화 결과 확인 | 시설·문제·지속성·정류장 근거 표시 |
| 3 | 중복·공식자료 대조 | 유사 신고와 BIT 대장 비교 | 최근 7일 4건, 공식 설치/시민 화면 꺼짐 충돌 설명 |
| 4 | AI 과업 권고 | 근거·기간·건수·불확실성 검토 후 승인/반려 | 유지보수 현장확인 과업 생성 |
| 5 | 현장조사 결과 | 전원 이상·수리 요청과 증거 확인 | 사람 확정 상태와 감사로그 기록 |
| 6 | 시민 화면 환류 | 변경 문구 미리보기 후 반영 | 시민 앱을 `버스정보안내기 점검 중`으로 갱신 |

관리자 화면은 모바일 카드가 아니라 넓은 표·지도·근거 패널을 사용하는 데스크톱 웹으로 구현한다.

## 핵심 화면 문구

- 첫 화면: `안녕하세요. 무엇을 도와드릴까요?`
- 버튼 1: `목적지로 가는 길 찾기`
- 버튼 2: `정류장 불편 알리기`
- 목적지 입력: `어디로 가세요?`
- 경로 계산: `탈 버스와 승차 정류장을 찾고 있어요`
- 결과 대표 카드: `300번 · 총 약 42분` / `버스 5분 후` / `타는 곳까지 도보 2분` / `10:42 도착 예상`
- 보조 행동: `버스를 탈 곳 보기`
- 결과 하단: `이 정류장에서 불편한 점 알리기`
- 신고 완료: `정류장 확인과 개선 검토 자료로 활용됩니다`

## 발표 데모 순서

1. 시민 앱에서 목적지 흐름 1→6을 시연한다.
2. 6번 결과에서 불편 신고 8번으로 연결한다.
3. 관리자 웹에서 같은 신고를 1→6으로 처리한다.
4. 시민 앱 결과가 `점검 중`으로 갱신되는 것을 다시 보여준다.

마지막 문장:

> **쉼표 정류장은 길을 알려주는 시민 앱과, 시민의 한마디를 검증 가능한 과업으로 바꾸는 관리자 웹을 하나의 데이터 순환으로 연결합니다.**
