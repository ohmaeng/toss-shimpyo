# 정류장 ID 내부 DB 작성 기준

## 원칙

- 원본 CSV는 열 이름과 값을 바꾸지 않고 보존한다.
- 내부 연결표는 `data/stop_id_mapping.csv` 한 파일을 기준으로 사용한다.
- 한 행은 `boarding_stop_id`(정산사업자 승차 정류장 ID) 하나를 뜻한다.
- 확인되지 않은 ID는 빈칸으로 둔다. `0`, `없음`, 임의의 정류장 번호를 넣지 않는다.
- 동일 ID에 원본 명칭이 여러 개면 삭제하지 않고 `|`로 함께 기록한다.

## 식별자

| 열 | 의미 |
|---|---|
| `boarding_stop_id` | 승하차 원본의 정산사업자 정류장 ID (`424…`) |
| `stop_no` | 이용자에게 표시되는 춘천시 4자리 정류장 번호 |
| `management_id` | 춘천시 위치정보의 관리번호 (`250…`, `257…`, `265…`) |

세 식별자는 서로 다른 기관·시스템의 키이므로 숫자가 비슷하다는 이유로 연결하지 않는다.

## 매칭 상태

| 값 | 사용 조건 |
|---|---|
| `direct` | 국토부 모바일아이디가 춘천시 4자리 번호와 직접 일치 |
| `inferred` | 명칭·법정동·현행 지도 등 둘 이상의 단서를 이용한 추정 연결 |
| `unresolved` | 직접 연결 근거가 없거나 후보가 둘 이상이라 미확정 |

`inferred`는 현장조사와 분석을 위한 임시 연결이며 공식 대응표를 받으면 `direct`로 교체한다.
승하차 원본 CSV의 행 배치는 노선 정차순서가 아니므로 전·후 행을 인접 정류장 근거로 사용하지 않는다.
노선정보를 이용한 자동 연결은 정류장명과 전체 경유노선 집합이 완전히 같고, 승차 ID와 관리번호가 양쪽 모두 1:1로 유일할 때만 허용한다.

## 수요 값

- `midday_boardings`는 승하차 원본에서 같은 `boarding_stop_id`의 11~16시 승차건수를 합산한다.
- 승차 원본에 해당 ID의 11~16시 행이 없거나 합계가 `0`이면 조사표와 내부 연결표에서는 빈칸으로 둔다. 원본의 `0`은 원본 CSV에 그대로 보존한다.
- 서로 다른 승차 ID는 정류장명이 같아도 합산하지 않는다.

## 재생성

```powershell
node pipeline\build_stop_id_mapping.mjs
```

수동 추정 결과는 `data/stop_id_mapping_overrides.csv`에만 기록한다. 생성된 CSV를 직접 고치지 않는다.
TAGO 경유노선 집합 교차검사 결과는 `data/stop_id_route_mapping_overrides.csv`, 판정 근거는 `data/stop_id_route_match_audit.csv`에서 관리한다.

## 조사표 표시 형식

- 개별 정류장과 공식 대응되지 않은 승차 ID를 우선순위 근거로 오해하지 않도록 현황조사표에서는 `승차자료 정류장 ID`, `표본기간 한낮 개별 승차건수`, `승차자료 매칭방법`, `승차자료 매칭신뢰등급` 열을 제외한다.
- 승차자료 원본과 내부 ID 대응 검증표는 공식 대응표 확보 후 재처리할 수 있도록 별도로 보존한다.
- `카카오맵 표시 방면`은 바깥 괄호를 제거하고 `정류장명 방면` 형식으로 통일한다.
- 카카오맵에서 방면을 확인하지 못한 경우 공식 노선순서 값을 복사하지 않고 빈칸으로 둔다.
- 공식 노선순서 기반 진행방면은 별도 열에 유지해 카카오맵 표시값과 섞지 않는다.
