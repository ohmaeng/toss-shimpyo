# 쉼표 정류장 — 사용자 흐름

> 파생 문서. 제품 결정은 `제품_기획.md`만 변경하며 이 문서는 화면·예외 흐름을 설명한다.

## 시민 앱 시작

```text
공통 QR/시민 앱
→ 안녕하세요. 무엇을 도와드릴까요?
   [목적지로 가는 길 찾기]
   [정류장 불편 알리기]
```

로그인·나이·현재 방면을 먼저 묻지 않는다.

## 목적지 길찾기

```text
[목적지로 가는 길 찾기]
→ “어디로 가세요?” 음성 또는 문자 입력
→ 인식한 목적지 확인
→ 현재 위치 + 목적지 + 노선정보로 경로 계산
→ 올바른 방향의 승차 정류장 자동 산출
→ 탈 수 있는 버스만 목적지 도착예상시간 순으로 정렬
→ 한 화면에 추천 버스 + 총 소요시간 + 버스 도착예정 + 타는 곳까지 도보시간
→ 목적지 도착예상 + 정류장명·4자리 번호·방면 + 대기환경
→ 선택: [이 정류장에서 불편한 점 알리기]
```

주변 정류장 선택 페이지와 별도 경로 상세 페이지는 두지 않는다. 총 소요시간은 `승차 정류장까지 도보 + 대기 + 버스 탑승 + 하차 후 도보`로 계산한다. 버스 도착예정시간이 `승차 정류장까지 도보시간 + 이동 여유시간`보다 짧은 후보는 제외한다. 결과 화면에서 산출된 승차 지점을 명확하게 보여주며, 위치 신뢰도가 낮거나 경로를 확정할 수 없을 때만 장소 직접 입력·재탐색 폴백을 제공한다.

## 불편 신고

```text
[정류장 불편 알리기]
→ 현재 위치 사용 또는 정류장명·4자리 번호 말하기/입력
→ 대상 정류장 확인
→ 음성 또는 큰 시설 버튼으로 불편 입력
→ AI가 시설·문제·지속성 후보 구조화
→ 사용자가 대상 정류장과 내용 확인·수정
→ 제출
→ 확인 절차 안내 + 필요 시 공식 민원 채널 이동
```

길찾기 결과에서 진입하면 이미 산출된 승차 정류장을 신고 대상으로 넘기고 다시 입력시키지 않는다.

## 도착·시설 안내 상태

- 실시간 도착: 제공기관과 생성시각 표시
- 대체 정보: 실제 성격을 표시하고 실시간이라 하지 않음
- API 실패: 도착정보를 숨기되 시설 안내와 불편 신고는 계속 이용
- 시설: `있음/없음 확인/미확인/정보 충돌`과 출처·확인일 표시
- 맞은편 시설: 산출된 승차 지점의 시설로 합치지 않음

## 관리자 웹

```text
대시보드
→ 신고 상세(원문 + AI 구조화)
→ 유사 신고 중복 묶음 + 공식자료 대조
→ AI 현장확인 과업 권고
→ 담당자 승인·반려·추가자료 요청
→ 현장조사 결과와 사람 확정
→ 시민 앱 시설상태 반영
```

AI 권고와 사람 확정은 라벨·색상·감사로그로 분리한다.
