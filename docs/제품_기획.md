# 쉼표 정류장 — 제품 기획 기준서

> 이 문서는 제품 정의의 단일 원본(SSOT)이다. 다른 기획 문서는 이 문서의 결정을 설명하거나 검증할 뿐, 별도의 제품 정의를 만들지 않는다.
> 1차 기획서의 `사계절 대기환경·시민 안내·행정 개선` 골조를 유지하면서 공공데이터 AI 활용 구조를 명확히 한다.

## 1. 제품 한 문장

**쉼표 정류장은 공통 QR로 실행한 시민 앱에서 목적지 음성 길찾기와 정류장 불편 신고를 제공하고, 시민이 말한 불편을 AI가 공공데이터와 대조해 정류장 단위의 현장확인 과업으로 바꾸는 시민–행정 순환 서비스다.**

도착정보는 시민 편익과 반복 이용을 위한 기능이다. 조회 기록을 실제 승차·하차·고령자 수요로 해석하지 않는다.

## 2. 해결할 문제

| 시민 문제 | 행정 문제 | 제품의 연결 |
|---|---|---|
| 버스가 언제 오는지와 어디서 기다릴지 한 번에 보기 어렵다 | 기관별 정류장 ID·명칭·시설상태·갱신시점이 다르다 | 현재 정류장의 도착·대기환경을 함께 안내 |
| 불편을 발견해도 정류장 식별과 민원 작성이 번거롭다 | 자유서술 제보의 중복 제거·시설 분류·근거 대조가 수작업이다 | 위치·정류장 확인 후 음성 또는 버튼으로 제보 |
| 제보 뒤 무엇이 바뀌었는지 알기 어렵다 | 미확인·고장·시설 요구가 섞여 조사 과업이 불명확하다 | AI 권고→사람 확인→시설정보 갱신→시민 화면 반영 |

## 3. 대상과 가치

- 1차 사용자: 폭염·한파와 긴 대기에 취약하고 복잡한 교통 앱 사용이 어려운 시민, 특히 고령 이용자
- 2차 사용자: 가족·요양보호사·복지기관 종사자와 일반 버스 이용자
- 행정 사용자: 춘천시 교통·시설·안전 담당자

`고령자 친화적`은 인터페이스의 설계 목표다. 로그인이나 자기응답이 없는 서비스 이벤트를 고령자 이용량으로 부르지 않는다. 연령별 실제 수요는 교통카드 운영기관의 익명 집계자료로만 보강한다.

## 4. 시민 핵심 흐름

```text
공통 QR 또는 시민 앱 실행
→ “무엇을 도와드릴까요?”
   ├─ 목적지로 가는 길 찾기
   │  → 목적지 음성 입력·확인
   │  → 현재 위치 + 목적지로 탈 버스와 올바른 방향의 승차 정류장 자동 산출
   │  → 한 화면에서 추천 버스·총 소요시간·도보·도착시간·대기환경 안내
   └─ 정류장 불편 알리기
      → 현재 위치 또는 정류장명·4자리 번호를 말하거나 입력
      → 대상 정류장 확인
      → 음성 또는 큰 버튼으로 불편 신고
      → AI 구조화 결과를 사용자가 확인 후 제출
```

공통 QR은 특정 정류장이나 방향을 내장하지 않는다. 목적지 흐름에는 주변 정류장 선택 페이지를 두지 않고 경로 엔진이 올바른 승차 지점을 산출한다. 불편 신고에서는 대상 오류를 막기 위해 위치 또는 정류장명·4자리 번호를 확인한다. 버스번호는 동일 노선이 여러 정류장을 지나므로 단독 식별자로 사용하지 않는다.

경로 엔진을 자체 개발하지 않는다. `길찾기_API_연계.md`에 따라 외부 대중교통 경로 API가 승차 정류장·노선·환승을 계산하고, 쉼표 정류장은 TAGO 실시간 도착정보와 춘천시 대기시설 정보를 검증해 결합한다.

## 5. 시민 화면의 필수 정보

1. 첫 화면의 `목적지로 가는 길 찾기`와 `정류장 불편 알리기`
2. 길찾기 결과의 정류장명, 4자리 번호, 방면, 승차 정류장까지 도보시간, 탈 버스
3. 버스별 총 소요시간·버스 도착예정·목적지 도착예상과 생성시각·실시간/대체정보 상태
4. 같은 승차 지점의 의자·그늘·쉘터·조명·버스정보안내기 상태
5. 시설별 출처·확인일과 `있음/없음 확인/미확인/정보 충돌`
6. 결과 화면 하단의 `이 정류장에서 불편한 점 알리기`

맞은편 시설과 수요를 합치거나 맞은편으로 건너가 기다리라고 권하지 않는다.

길찾기 결과는 별도 상세 경로 화면으로 나누지 않는다. 첫 카드에는 `300번 · 총 약 42분`, `버스 5분 후`, `타는 곳까지 도보 2분`, `10:42 도착 예상`처럼 한 화면에서 판단할 정보를 제공한다. 추천 순서는 버스가 정류장에 빨리 오는 순서가 아니라 `승차 정류장까지 도보 + 대기 + 버스 탑승 + 하차 후 도보`를 합친 목적지 도착예상시간 순서다. 사용자의 도보시간과 이동 여유시간보다 먼저 도착하는 버스는 탈 수 없는 후보로 제외한다.

## 6. 불편 알리기

사용자에게는 하나의 `불편 알리기`만 제공한다. 자체 서비스의 제보를 정식 민원이라고 부르지 않는다.

| 시민 표현 | 내부 분류 | 행정 의미 |
|---|---|---|
| 앉을 곳이 없어요 | 시설 필요 의견 | 개선 검토 신호 |
| 햇빛·비를 피하기 어려워요 | 그늘/쉘터 필요 의견 | 계절별 현장확인 신호 |
| 너무 어두워요 | 조명 문제 | 안전·유지보수 확인 |
| 버스 도착 안내 화면이 안 나와요 | BIT 작동 불량 | 설치대장 대조·유지보수 확인 |
| 표시된 시설정보가 실제와 달라요 | 시설정보 불일치 | 데이터 재확인 |

사실 주장(`없어졌어요`, `고장 났어요`)과 정책 요구(`필요해요`, `불편해요`)는 데이터베이스에서 분리한다. 공식 답변이 필요한 이용자에게만 제출 완료 화면에서 춘천시의 기존 공식 민원 채널을 안내한다. 공식 연계 전에는 `춘천시에 바로 접수`라고 표시하지 않는다.

## 7. AI가 반드시 필요한 지점

AI는 장식용 요약기가 아니라 비정형 시민 발화와 서로 다른 공공자료를 행정이 검토할 수 있는 단위로 바꾸는 핵심 엔진이다.

```text
시민 음성·선택형 신호 + 공공 정류장·노선·BIT·시설자료
→ 발화 구조화(시설·문제·지속성·긴급 표현)
→ 정류장 후보 및 데이터 충돌 탐지
→ 유사 제보 중복 묶기·반복 패턴 탐지
→ 데이터 정정/유지보수/현장조사/개선 검토로 과업 분류
→ 출처·기간·건수·불확실성을 포함한 권고안 생성
→ 담당자 승인·반려·현장 확인
```

확정 ID 일치와 공식 수치 집계는 규칙·SQL로 수행한다. AI는 애매한 정류장 매칭을 확정하거나 시설 상태·공사·예산을 최종 결정하지 않는다.

## 8. 행정 순환

1. AI가 시민 발화를 구조화하고 동일 정류장·문제의 중복을 묶는다.
2. 공식 설치대장·시설 관측과 비교해 `정보 충돌`, `고장 가능성`, `시설 요구`를 구별한다.
3. 근거가 포함된 현장확인 권고를 만든다.
4. 담당자가 승인·반려하거나 현장조사를 배정한다.
5. 사람이 확인한 결과만 시설 관측이력과 운영 상태를 변경한다.
6. 최신 상태가 시민 화면에 다시 반영된다.

성과지표는 제보 수가 아니라 `중복 제거 후 유효 후보`, `실제 불일치 확인률`, `건당 검토시간`, `갱신 소요기간`, `시민 화면 재반영률`로 잡는다.

## 9. 데이터 원칙

- 공식: 춘천시·국토교통부·TAGO·BIT 자료
- 확정 관측: 현장조사 및 담당자 검수 결과
- 참고: 카카오맵 등 민간 지도 교차검토
- 검수 전 신호: 시민 제보와 서비스 이벤트
- 시설 상태: `있음/없음 확인/미확인/정보 충돌`
- 분석 단위: 맞은편과 합치지 않은 개별 승차 지점
- 정류장 연결: 공식 ID 직접 일치 또는 좌표·노선·정차순서·방면의 복수 근거가 있는 경우만 확정

시민 이벤트는 `승차예정 정류장 기반 조회·불편 신호`다. 실제 승차, 고령자 수요, 수혜 인원으로 표현하지 않는다.

## 10. 개인정보 최소화

- 경로 계산에 사용한 정확한 GPS 원본은 서버에 장기 저장하지 않는다.
- 서버에는 사용자가 확인한 정류장 ID, 넓은 시간구간, 행동, 구조화된 불편유형만 저장한다.
- 이름·전화번호·전체 생년월일·집 주소·광고 ID·지속 위치추적을 사용하지 않는다.
- 목적지는 경로 계산에 사용하되 상세 검색어의 장기 보관을 기본값으로 하지 않는다. 행정 집계가 필요하면 별도 동의 후 넓은 목적지 범주만 사용한다.
- 원시 이벤트는 짧게 보관하고 주·월 단위로 집계하며 소수 조합은 공개하지 않는다.
- 길 안내와 선택적 데이터 제공 동의를 분리한다.

## 11. 본선 핵심 범위와 확장

### P0 — 반드시 한 사이클로 시연

1. 공통 QR 진입과 첫 화면의 두 업무 선택
2. 목적지 음성 입력·확인과 승차 정류장 자동 산출
3. 버스·도착정보·대기환경 안내
4. 별도 진입 또는 결과 화면에서 음성/버튼 불편 신고와 AI 구조화
5. 관리자 웹의 공식자료 대조·중복 묶기·현장확인 권고
6. 담당자 확정 후 시민 앱의 시설상태 갱신

### P1 — 완성도를 높이는 요소

- TTS와 큰 글자·고대비·음성 실패 폴백
- 정류장 데이터 충돌 근거 카드
- 조사 과업표 CSV/A4 출력
- 제출 이후 `확인 중/확인 완료` 상태 공개

### P2 — 협약·공식자료 확보 후 확장

- 춘천시 공식 민원 시스템 양식 연계
- 방향별 시간·연령유형 교통카드 집계 연계
- 목적지·경로 즐겨찾기
- 겨울 한파·야간 조명·재난 대응 프리셋

## 12. 중단 조건

- AI 추정 정류장 매칭을 공식 대응이라고 표시함
- 도착정보 조회를 실제 승차라고 표시함
- 시민 제보량을 고령자 수요나 설치순위로 사용함
- 미확인을 없음으로 바꿈
- 카카오맵 정보를 공공기관 원본과 같은 지위로 사용함
- AI가 시설 상태·공사·예산을 자동 확정함
- 공식 연계 없이 정식 민원이 접수됐다고 표시함

## 13. 제품 결정의 원자성

- 제품 정의 변경은 이 문서에서 먼저 한다.
- 같은 변경에서 `사용자_흐름`, `본선_스토리보드`, `본선_전략`, `사용자_테스트`를 함께 갱신한다.
- 데이터 필드와 보관정책은 `데이터_AI_계약.md`에서만 상세화한다.
- 과거 계획 문서는 역사 기록이며 현재 요구사항으로 사용하지 않는다.
- 현재 구현과 목표 기획을 구분하고, 미구현 기능을 완료된 것처럼 발표하지 않는다.
